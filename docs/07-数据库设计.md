# 第 7 篇：数据库设计——每张表、每个字段、每个索引为什么这样设计

---

## 一、数据库概览

本项目共 **10 张表**（原 v2 有 8 张，v3 新增 2 张）：

| 表名 | 用途 | v2/v3 |
|------|------|-------|
| users | 用户 | v2 升级 |
| categories | 新闻分类 | v2 |
| news | 新闻稿件（核心表） | v2 升级 |
| reviews | 评论（嵌套回复） | v2 |
| slides | 轮播图 | v2（v3 中轮播改用置顶新闻） |
| logs | 操作日志 | v2 |
| favorites | 用户收藏 | v2 |
| likes | 用户点赞 | v2 |
| read_history | 浏览历史 | v2 |
| notifications | 实时通知 | **v3 新增** |
| user_behavior | 用户行为（推荐数据源） | **v3 新增** |

## 二、每张表详解

### 2.1 users —— 用户表

```sql
CREATE TABLE users (
    user_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    username   VARCHAR(50) NOT NULL UNIQUE,    -- 用户名，唯一
    password   VARCHAR(255) NOT NULL,           -- BCrypt 哈希（v2: VARCHAR(32) MD5）
    email      VARCHAR(100) DEFAULT NULL,       -- v3 新增
    phone      VARCHAR(20) DEFAULT NULL,        -- v3 新增
    role       ENUM('admin','auditor','editor','user') NOT NULL DEFAULT 'editor',
    avatar     VARCHAR(255) DEFAULT NULL,       -- 头像路径
    status     TINYINT(1) DEFAULT 1,            -- 1=启用 0=停用
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**字段设计理由**：

| 设计决策 | 理由 |
|----------|------|
| `password` VARCHAR(255) | BCrypt 哈希固定 60 字符，255 留足余量（v2 的 32 只够 MD5） |
| `role` 用 ENUM | 角色固定 4 种，ENUM 比 VARCHAR 省空间、有约束、查询快 |
| `status` TINYINT | 布尔值用 TINYINT 不用 BOOLEAN（MySQL BOOLEAN 本质是 TINYINT） |
| `email` 可 NULL | 注册时选填，不强制 |
| `created_at` DEFAULT CURRENT_TIMESTAMP | 自动记录注册时间，不用代码设置 |

**v2 → v3 变更**：
- password 从 VARCHAR(32) 扩到 VARCHAR(255)（MD5 → BCrypt）
- 新增 email、phone 字段
- 字符集从 utf8 升级到 utf8mb4（支持 emoji）

### 2.2 categories —— 新闻分类表

```sql
CREATE TABLE categories (
    category_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(30) NOT NULL,
    sort_order  INT DEFAULT 0,          -- 排序（决定首页导航显示顺序）
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

种子数据：体育(1)、财经(2)、科技(3)、文化(4)、时政(5)、教育(6)、民生(7)

**sort_order 的作用**：首页导航和侧边栏按这个字段排序，管理员可以调整顺序。

### 2.3 news —— 新闻表（核心）

```sql
CREATE TABLE news (
    news_id       BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       BIGINT NOT NULL,              -- 作者
    category_id   BIGINT NOT NULL,              -- 所属分类
    title         VARCHAR(100) NOT NULL,        -- 标题
    summary       VARCHAR(255) DEFAULT NULL,    -- 摘要（v3 新增）
    content       TEXT,                         -- 正文
    tags          VARCHAR(255) DEFAULT NULL,    -- 标签（v3 新增，逗号分隔）
    source        VARCHAR(100) DEFAULT NULL,    -- 来源（v3 新增）
    cover_image   VARCHAR(255) DEFAULT NULL,    -- 封面图路径
    status        ENUM('draft','pending','published','rejected') DEFAULT 'draft',
    reject_reason VARCHAR(255) DEFAULT NULL,    -- 退回原因
    is_top        TINYINT(1) DEFAULT 0,         -- 是否置顶（v3：置顶=轮播）
    clicked       BIGINT DEFAULT 0,             -- 点击量
    is_es_synced  TINYINT(1) DEFAULT 0,         -- ES 同步标记（v3 新增）
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (category_id) REFERENCES categories(category_id),

    INDEX idx_status_cat_time (status, category_id, created_at),  -- v3 新增
    INDEX idx_user_status (user_id, status)                        -- v3 新增
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**status 状态机**：
```
draft ──→ pending ──→ published
  ↑                    (审核通过)
  │         └──→ rejected （审核退回）
  └──────────────┘（修改后重新提交）
```

**索引设计理由**：

| 索引 | 覆盖查询 | 为什么 |
|------|---------|--------|
| `idx_status_cat_time` | `WHERE status='published' AND category_id=1 ORDER BY created_at DESC` | 首页分类筛选是最频繁的查询，联合索引三个字段覆盖完整查询 |
| `idx_user_status` | `WHERE user_id=3 AND status='pending'` | "我的稿件"按状态筛选，编辑最常用的查询 |

**v2 → v3 变更**：
- 新增 summary、tags、source 字段
- clicked 从 INT 改为 BIGINT（支持过亿点击）
- 新增两个联合索引
- `is_top` 在 v3 中有新含义：置顶 = 出现在首页轮播

### 2.4 reviews —— 评论表（嵌套回复）

```sql
CREATE TABLE reviews (
    review_id  BIGINT AUTO_INCREMENT PRIMARY KEY,
    news_id    BIGINT NOT NULL,              -- 属于哪篇新闻
    user_id    BIGINT DEFAULT NULL,          -- 评论者（NULL=匿名）
    parent_id  BIGINT DEFAULT NULL,          -- 父评论ID（NULL=顶级评论）
    username   VARCHAR(50) DEFAULT '匿名',   -- 显示用的用户名
    content    TEXT NOT NULL,
    state      ENUM('pending','approved') DEFAULT 'approved',
    ip         VARCHAR(45) DEFAULT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE,
    FOREIGN KEY (parent_id) REFERENCES reviews(review_id) ON DELETE CASCADE,

    INDEX idx_news_state (news_id, state, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**嵌套回复怎么实现的？**
```
顶级评论A (parent_id=NULL)
  ├── 回复A1 (parent_id=A.review_id)
  └── 回复A2 (parent_id=A.review_id)
顶级评论B (parent_id=NULL)
  └── 回复B1 (parent_id=B.review_id)
```

查询时先查 `parent_id IS NULL` 的顶级评论，再查 `parent_id IN (...)` 的子回复。后端 ReviewController 在内存中组装成嵌套结构。

**为什么用 parent_id 而不是另建表？**
- 一张表实现无限层级嵌套
- 不用额外的关联表
- 查询简单：两次 SELECT 即可

**ON DELETE CASCADE**：新闻被删除时，它的所有评论自动删除。父评论被删除时，子回复也删除。

**为什么 IP 是 VARCHAR(45)？** IPv4 最长 15 字符（如 192.168.1.1），IPv6 最长 45 字符。

### 2.5 slides —— 轮播图表

```sql
CREATE TABLE slides (
    slide_id   BIGINT AUTO_INCREMENT PRIMARY KEY,
    title      VARCHAR(100) DEFAULT NULL,
    image      VARCHAR(255) NOT NULL,        -- 图片路径
    link_url   VARCHAR(255) DEFAULT NULL,    -- 点击跳转链接
    sort_order INT DEFAULT 0,
    status     TINYINT(1) DEFAULT 1,         -- 1=启用 0=停用
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**v3 中的变化**：这张表仍然存在，但首页轮播已改用置顶新闻。这个表作为备用（管理员仍可通过后台管理独立轮播）。

### 2.6 logs —— 操作日志表

```sql
CREATE TABLE logs (
    log_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id   BIGINT DEFAULT NULL,
    username  VARCHAR(50) DEFAULT NULL,
    action    VARCHAR(100) NOT NULL,          -- 操作描述（如"登录系统""删除稿件"）
    target    VARCHAR(100) DEFAULT NULL,      -- 操作目标（如"稿件#123"）
    ip        VARCHAR(45) DEFAULT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_created_at (created_at)         -- v3 新增：按时间查询日志
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**为什么不用外键关联 users 表？** 用户可能被删除，但日志应该保留。所以冗余了 username 字段，即使删了用户也能看到"谁做了什么"。

**AOP 如何写日志？** LogAspect 切面拦截所有带 `@LogOperation` 的方法，从中获取当前用户和操作描述，异步写入 logs 表。

### 2.7 favorites —— 收藏表

```sql
CREATE TABLE favorites (
    fav_id     BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    news_id    BIGINT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_user_news (user_id, news_id),  -- 同一用户不能重复收藏
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**为什么单独建表而不是在 users 表加字段？** 收藏是多对多关系（一个用户收藏多篇新闻，一篇新闻被多人收藏）。关系必然要单独建表。

**UNIQUE KEY**：保证不会重复收藏。如果 INSERT 时违反唯一约束，MyBatis-Plus 会抛异常。

---

### 2.8 likes —— 点赞表

```sql
CREATE TABLE likes (
    like_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    news_id    BIGINT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_user_news (user_id, news_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**为什么不把点赞数存在 news 表里？** 点赞是用户行为，news 表应该只存新闻本身的信息。点赞数可以通过 `SELECT COUNT(*) FROM likes WHERE news_id=?` 实时计算，也可以通过定时任务更新缓存。

---

### 2.9 read_history —— 浏览历史

```sql
CREATE TABLE read_history (
    hist_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    news_id    BIGINT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_user_news (user_id, news_id),   -- 同一篇新闻不重复记录
    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE,
    INDEX idx_user_time (user_id, created_at)      -- v3 新增
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**UNIQUE KEY 的作用**：同一用户看同一篇新闻多次，只保留最新的记录。如果已存在 → 更新 created_at 即可。

---

### 2.10 notifications —— 通知表（v3 新增）

```sql
CREATE TABLE notifications (
    notif_id   BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id    BIGINT NOT NULL,                  -- 接收通知的用户
    type       ENUM('approved','rejected','comment','system') NOT NULL,
    title      VARCHAR(200) NOT NULL,            -- 通知标题
    content    VARCHAR(500) DEFAULT NULL,        -- 通知内容
    target_id  BIGINT DEFAULT NULL,              -- 关联的 news_id 或 review_id
    is_read    TINYINT(1) DEFAULT 0,             -- 0=未读 1=已读
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    INDEX idx_user_read (user_id, is_read, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**type 的四种值**：
- `approved`：稿件审核通过
- `rejected`：稿件被退回
- `comment`：有人评论了你的稿件
- `system`：系统通知（如管理员广播）

**idx_user_read**：覆盖"查我的未读通知"这个最常见查询。

### 2.11 user_behavior —— 用户行为表（v3 新增）

```sql
CREATE TABLE user_behavior (
    behavior_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id        BIGINT NOT NULL,
    news_id        BIGINT NOT NULL,
    behavior_type  ENUM('view','like','favorite','comment') NOT NULL,
    behavior_score DECIMAL(3,2) DEFAULT 1.00,  -- 行为权重：view(0.3), like(0.7), favorite(1.0), comment(0.5)
    created_at     DATETIME DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE,
    INDEX idx_user_behavior (user_id, behavior_type, created_at),
    INDEX idx_news_behavior (news_id, behavior_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**为什么需要这张表？** 这是协同过滤推荐的数据源。不同行为有不同的权重：
- 浏览(view)：0.3 分（看过不一定喜欢）
- 点赞(like)：0.7 分（喜欢才会点）
- 收藏(favorite)：1.0 分（非常喜欢才收藏）
- 评论(comment)：0.5 分（可能是负面评论）

**目前的状态**：表已建好，代码还没写入（推荐系统目前用的是基于内容的 TF-IDF）。面试时可以讲"数据模型已建好，后续可以升级为混合推荐"。

## 三、为什么用 InnoDB 而不是 MyISAM？

| | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | ✅ 支持 | ❌ 不支持 |
| 外键 | ✅ 支持 | ❌ 不支持 |
| 行级锁 | ✅ | ❌（表级锁） |
| 崩溃恢复 | ✅ | ❌ |
| 适用场景 | 需要数据完整性的业务系统 | 只读/日志归档 |

本项目有审核流、点赞收藏等写操作，必须用 InnoDB。

---

## 四、为什么用 utf8mb4 而不是 utf8？

MySQL 的 `utf8` 实际上是阉割版，每个字符最多 3 字节，不支持 emoji（4 字节）。`utf8mb4` 是完整版，支持所有 Unicode 字符。

---

## 五、外键的取舍

**本项目中用了外键**，但实际生产环境大厂经常**不用外键**，而是在应用层保证数据一致性。

| | 用外键 | 不用外键 |
|------|--------|---------|
| 数据一致性 | 数据库保证 | 应用层保证 |
| 性能 | 写操作需检查约束，稍慢 | 更快 |
| 灵活性 | 删数据受约束 | 自由 |
| 分库分表 | 无法跨库 | 可以 |

**面试时可以说**："项目中用了外键保证数据完整性，适合中小规模。如果数据量大到需要分库分表，外键会变成瓶颈，那时候在应用层用逻辑外键。"

## 六、ER 图（实体关系图）

```
users ──1:N──→ news ──1:N──→ reviews
  │              │               │
  │              │               └── parent_id (自引用)
  │              │
  │              ├── N:1 ──→ categories
  │              │
  │              └── 1:N ──→ favorites
  │              └── 1:N ──→ likes
  │              └── 1:N ──→ read_history
  │              └── 1:N ──→ user_behavior
  │
  ├── 1:N ──→ logs
  ├── 1:N ──→ notifications
  └── 1:N ──→ slides (弱关联)
```
