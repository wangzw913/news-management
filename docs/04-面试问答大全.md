## 一、项目整体（必问）

### Q1: 简单介绍一下你的项目

**标准回答**：
"我做的是一个新闻发布管理系统，支持四种角色协作：普通用户浏览新闻、收藏点赞评论投稿，编辑写稿提交审核，审核员审核稿件，管理员管理整个系统。技术上从 PHP 原生版本升级到了 Spring Boot + Vue 3 全栈架构，新增了内容推荐算法、WebSocket 实时通知、Redis 缓存、JWT 双 token 认证这些功能。"

**你的底层逻辑**：30 秒内讲清楚"是什么 + 技术栈 + 亮点"

### Q2: 为什么把 PHP 改成 Spring Boot？

**标准回答**：
"有三个原因。第一，PHP 版本的代码是面向过程的，所有逻辑混在一起，扩展和维护困难。Spring Boot 的分层架构让 Controller、Service、Mapper 职责清晰。第二，我加入了推荐算法和实时通知这些需要复杂后端逻辑的功能，PHP 不太适合。第三，保研面试中 Java + Spring Boot 是主流，导师更熟悉，能问出有深度的问题。"

### Q3: 前后端怎么通信的？

**标准回答**：
"前端用 Axios 发送 HTTP 请求，后端暴露 RESTful API。所有接口路径统一是 /api/v1/，返回 JSON 格式。前端配置了 Vite 代理，开发时把 /api 请求转发到 8080 端口，解决跨域问题。后端也配了 CORS 允许跨域。生产环境用 Nginx 反向代理，前端和后端都走同一个 80 端口。"

**追问**："什么是跨域？怎么解决？"
→ "浏览器同源策略：协议、域名、端口任何一个不同就是跨域。localhost:5173 和 localhost:8080 就是跨域。解决方案：开发时 Vite 代理，生产时 Nginx 反向代理，后端加 CORS 配置做兜底。"

### Q4: 你这个项目有什么亮点/难点？

**标准回答**：
"有四个亮点。第一是内容推荐系统，基于 TF-IDF 算法做中文分词和余弦相似度计算。第二是点击计数的 Write-Behind 优化，用内存缓冲替代每次直接写数据库。第三是 JWT 双 token 认证，access token 30 分钟过期，refresh token 7 天，登出时 token 加入黑名单。第四是 Docker Compose 一键部署 6 个服务。"

### Q5: 项目架构怎么设计的？为什么这样分层？

**标准回答**：
"后端分四层。Controller 只接收请求和返回结果，Service 处理所有业务逻辑，Mapper 负责数据库操作。数据对象也分三层：Entity 映射数据库表，DTO 接收前端参数，VO 返回给前端。这样分层的好处是代码职责清晰，方便单元测试，也方便多人协作。"

### Q6: 你做的测试覆盖了哪些场景？

**标准回答**：
"我写了 Service 层的单元测试，用 JUnit 5 加 Mockito，Mock 了 Mapper 层来隔离数据库。UserService 的测试覆盖了 6 个场景：登录成功、密码错误、用户不存在、账号停用、注册成功、用户名重复。每个测试都验证了正常路径和异常路径。"

## 二、认证与安全（高频）

### Q7: JWT 和 Session 有什么区别？为什么选 JWT？

| | Session | JWT |
|------|--------|-----|
| 状态存储 | 服务端（内存/Redis） | 客户端（token 本身） |
| 扩展性 | 需要共享 Session（粘性会话或 Redis） | 天然无状态，任意服务器都能验证 |
| 注销 | 直接删 Session | 需要黑名单机制 |
| 适用场景 | 单体应用 | 分布式/微服务 |

"我选 JWT 因为前后端分离 + 无状态方便扩展。但 JWT 有个缺点——签发了就无法撤销。所以用短有效期（30分钟）+ 黑名单机制解决。"

### Q8: 双 token 怎么设计的？为什么这样设计？

**标准回答**：
"Access Token 有效期 30 分钟，每次请求都带。Refresh Token 有效期 7 天，存在 Redis 里，Access Token 过期后用 Refresh Token 换新的。这样设计的好处：Access Token 短 → 即使被盗 30 分钟后就失效；Refresh Token 长 → 不用频繁登录。还有一个细节：Refresh Token 是'一次一换'的，每次使用后旧的立即失效，换一个新的，防止重放攻击。"

**追问**："如果 Refresh Token 也被盗了呢？"
→ "这种情况说明客户端设备已经被完全控制。但我们的 Refresh Token 一次一换机制可以检测到：如果攻击者和原主同时使用同一个 Refresh Token，其中一个人会失败，系统可以发通知提醒用户。"

### Q9: BCrypt 比 MD5 好在哪里？

**标准回答**：
"MD5 有三个致命缺陷：相同的密码产生相同的哈希 → 彩虹表能批量破解；计算速度极快 → 暴力破解几秒钟试几十万次；没有盐值。BCrypt 解决了这三个问题：自动生成随机盐值，相同密码每次结果不同；计算速度故意慢（cost factor=10，约 100ms），暴力破解成本极高。现在是业界标准，Spring Security 默认推荐。"

### Q10: 你的注册接口安全吗？怎么防止恶意注册？

**标准回答**：
"有三层防护。第一，注册强制为普通用户角色，即使有人伪造请求传 role=admin，后端也强制设置为 user。第二，加了 @RateLimit 限流，同一用户 60 秒内最多注册 5 次。第三，密码长度 6-50 位限制。生产环境还可以加邮箱验证码和图形验证码，因为需要 SMTP 服务器所以开发环境没做。"

### Q11: Spring Security 的过滤器链是什么？

**标准回答**：
"过滤器链是一串按顺序执行的过滤器，每个过滤器做一件事。我的项目里，JwtAuthenticationFilter 在 UsernamePasswordAuthenticationFilter 之前执行。它从请求头提取 token，验证有效性，解析出用户信息，设到 SecurityContext 里。然后 SecurityFilterChain 根据 URL 规则检查权限。"

## 三、数据库与 MyBatis-Plus

### Q12: 为什么用 MyBatis-Plus 而不是 JPA？

**标准回答**：
"MyBatis-Plus 更灵活，复杂查询可以直接写 SQL。JPA 自动生成 SQL，简单的省事，但复杂查询（多表 JOIN、聚合统计）很难优化。国内公司普遍用 MyBatis，面试官更熟悉。另外 MyBatis-Plus 自带分页、自动填充、乐观锁，开发效率不输 JPA。"

### Q13: Entity、DTO、VO 为什么要分开？

**标准回答**：
"安全和灵活。Entity 有 password 字段，但 UserVO 返回给前端时绝对不能包含密码。LoginDTO 需要 username+password，但 LoginResultVO 返回 token+角色。如果共用同一个对象，字段混在一起很容易出安全问题。分开后每个场景用专门的类，字段清晰，不会泄露敏感数据。"

### Q14: 审核工作流的状态是怎么设计的？

**标准回答**：
"用状态机：草稿(draft) → 待审核(pending) → 已发布(published) 或 已退回(rejected)。退回后作者可以修改再提交，又回到 pending。每个状态转移都有校验，比如已发布的不能退回，已通过的不能再次提交。状态是数据库的 ENUM 字段，MyBatis-Plus 自动映射成字符串。"

### Q15: 分页是怎么实现的？

**标准回答**：
"MyBatis-Plus 自带分页插件。在 MyBatisPlusConfig 里注册了 PaginationInnerInterceptor，之后所有的 selectPage 调用会自动在 SQL 后面加 LIMIT。前端传 page 和 size 参数，后端返回 records（当前页数据）+ total（总条数），前端 Element Plus 的 Pagination 组件直接渲染。"

### Q16: 数据库索引你加了哪些？为什么？

**标准回答**：
"加了四个关键索引。news 表的 (status, category_id, created_at) 联合索引覆盖了最常用的筛选查询——按状态和分类查新闻列表并按时间排序。user_id 和 status 的联合索引加速'我的投稿'查询。reviews 表的 (news_id, state, created_at) 加速评论列表。logs 表的 created_at 索引加速日志查询。这些都是根据实际查询场景加的，不是盲目建索引。"

## 四、推荐算法（加分关键）

### Q17: 你的推荐系统怎么做的？

**标准回答**：
"用的是基于内容的推荐。步骤如下：第一步，用 Unigram+Bigram 对中文文本做简单分词；第二步，对每个词计算 TF-IDF 值形成文档向量；第三步，用余弦相似度计算当前文章和其他文章的相似度；第四步，取 Top5 返回。另外有 LRU 缓存避免重复计算，冷启动时降级到同分类推荐。"

### Q18: TF-IDF 是什么？怎么算的？

**标准回答**：
"TF 是词频，一个词在文档中出现次数除以文档总词数，衡量这个词对本文有多重要。IDF 是逆文档频率，总文档数除以包含该词的文档数的对数，衡量这个词的稀有度。TF 乘 IDF 就是最终的权重。一个词在本文出现多但在其他文档出现少 → TF-IDF 值高 → 对本文很重要。"

### Q19: 余弦相似度是什么？

**标准回答**：
"余弦相似度衡量两个向量方向的一致性。公式是 A·B / (|A|×|B|)。两个文档向量夹角越小（余弦值越接近 1），内容越相似。比如一篇讲 AI 的文章和一篇讲机器学习的文章，它们的 TF-IDF 向量方向接近，余弦相似度高。"

### Q20: 冷启动问题怎么解决？

**标准回答**：
"基于内容的推荐天然冷启动友好，因为不需要用户行为数据，只要有文章内容就行。如果文章内容太短（少于 100 字），TF-IDF 效果不好，就降级到同分类高点击推荐。后续用户行为数据积累多了，可以加上协同过滤。"

**追问**："协同过滤和内容推荐有什么区别？"
→ "内容推荐基于文章本身的特征（标题、正文），不管用户行为。协同过滤基于'相似用户喜欢相似内容'的假设，需要大量用户行为数据。我的项目两者结合：内容推荐做兜底，行为数据多了就上协同过滤。"

## 五、缓存与性能（深度话题）

### Q21: 点击计数为什么不直接 UPDATE？

**标准回答**：
"直接 UPDATE 有两个问题。第一是行锁竞争：高并发时多个请求同时 UPDATE 同一行，MySQL 加行锁，后来的请求排队等。第二是写放大：每次阅读都触发一次磁盘写入。我的方案是 Write-Behind 模式：先用 ConcurrentHashMap 在内存累加计数，每 5 分钟通过 @Scheduled 定时任务批量 UPDATE 一次，把 1000 次写变成 1 次写。"

### Q22: 缓存穿透、击穿、雪崩分别是什么？

| 问题 | 定义 | 本项目的解法 |
|------|------|------------|
| 穿透 | 查不存在的数据，每次穿透到 DB | 空值也缓存（短 TTL） |
| 击穿 | 热点数据过期瞬间，大量请求打 DB | 分布式锁（SETNX），拿到锁的去查 DB，其他的等 |
| 雪崩 | 大量缓存同时过期，DB 瞬间崩溃 | 随机 TTL（在基础 TTL 上加随机偏移） |

### Q23: 分布式锁怎么实现的？

**标准回答**：
"用 Redis 的 SETNX 命令。SET key value NX EX timeout，NX 表示只在 key 不存在时设置（互斥），EX 是过期时间（防死锁）。拿到锁的线程去查数据库重建缓存，没拿到的等 100ms 再重试或直接返回旧缓存。"

## 六、部署与 DevOps

### Q24: Docker Compose 里有哪些服务？为什么这样拆分？

**标准回答**：
"6 个服务：MySQL（数据库）、Redis（缓存）、Elasticsearch（搜索）、MinIO（文件存储）、Spring Boot（后端）、Nginx+Vue（前端）。拆分原则是单一职责：每个容器只做一件事，方便独立扩缩容和故障隔离。比如搜索压力大时，可以单独扩容 ES，不影响其他服务。"

### Q25: Dockerfile 为什么用多阶段构建？

**标准回答**：
"第一阶段用 Maven 镜像编译打包（包含 JDK 和 Maven），第二阶段只用 JRE 运行（不包含 Maven 和源码）。最终镜像体积从 500MB+ 降到 150MB，启动更快，也更安全（攻击面小）。"

### Q26: Nginx 在你的项目里做什么？

**标准回答**：
"两个作用。第一是静态资源服务器，托管 Vue 编译后的 HTML/CSS/JS。第二是反向代理，把 /api/ 路径的请求转发到 Spring Boot 后端（8080 端口），/ws/ 路径的 WebSocket 请求也代理过去。这样前端后端都走同一个 80 端口，不存在跨域问题。"

### Q27: CI/CD 流水线做了什么？

**标准回答**：
"GitHub Actions 自动触发。每次 push 代码，自动跑后端测试（mvn test）、前端构建（npm run build）、打 Docker 镜像。如果测试失败，合并被阻止。生产环境可以加上自动部署：SSH 到云服务器 docker compose pull && up -d。"

## 七、测试（导师必问）

### Q28: 你写了哪些测试？怎么写的？

**标准回答**：
"写了 UserService 的单元测试，6 个用例覆盖登录和注册的正常和异常场景。用 JUnit 5 做测试框架，Mockito Mock 了 UserMapper 和 PasswordEncoder。比如测'密码错误'时，Mock passwordEncoder.matches 返回 false，然后断言抛出了 BusinessException。因为 MyBatis-Plus 的 ServiceImpl 需要 baseMapper，我用 ReflectionTestUtils 手动注入了 Mock Mapper。"

### Q29: 为什么 Mock 而不连接真实数据库？

**标准回答**：
"单元测试只测业务逻辑，不测数据库。Mock 隔离了外部依赖，测试跑得快（毫秒级），不需要启动 MySQL。如果连真实数据库，测试变慢（秒级），而且测试数据会影响其他测试。集成测试才连真实数据库，那是另一个测试层面。"

### Q30: 如果继续完善测试，你还会加什么？

**标准回答**：
"第一，Service 层的集成测试：用 @SpringBootTest + H2 内存数据库，测完整的 Service → Mapper → DB 链路。第二，Controller 层的 API 测试：用 MockMvc 发 HTTP 请求，验证返回的 JSON 结构。第三，前端 Pinia Store 的单元测试。"

## 八、前端相关

### Q31: Vue 的响应式原理是什么？

**标准回答**：
"Vue 3 用 Proxy 拦截对象属性的读取和修改。当你 `const count = ref(0)` 时，Vue 创建了一个响应式引用。读取 count.value 时，Vue 跟踪是哪个组件在读；修改 count.value 时，Vue 通知那个组件重新渲染。整个过程是自动的，开发者不需要手动操作 DOM。"

### Q32: 路由守卫做了什么？

**标准回答**：
"beforeEach 守卫在每次页面跳转前执行。逻辑是：公开页面直接放行 → 未登录跳登录页 → 有 token 但无用户信息先加载 → 已登录但访问登录页按角色重定向 → 角色不匹配跳 403。这样保证用户不会看到不该看到的页面，也不会因为 token 过期而卡住。"

### Q33: Pinia 状态管理和 localStorage 怎么配合的？

**标准回答**：
"Pinia 的 persist 插件自动把状态存 localStorage，页面刷新后自动恢复。我选了 accessToken、refreshToken、userId、username、role 这几个关键字段做持久化。这样用户刷新页面不需要重新登录。敏感信息（如密码）不持久化。"

### Q34: Axios 拦截器做了什么？

**标准回答**：
"请求拦截器自动在每个请求头加 Authorization: Bearer token。响应拦截器做了三件事：自动解包后端返回的 {code, message, data} 结构，只返回 data；遇到 401 自动用 Refresh Token 刷新 Access Token；其他错误自动弹提示。这些是全局的，不用每个 API 调用都写 try-catch。"

## 九、安全与扩展（深度加分）

### Q35: 如果有人伪造 JWT token 怎么办？

**标准回答**：
"伪造不了。JWT 第三部分是签名，用服务端的密钥通过 HMAC-SHA384 计算。攻击者改了 payload，签名就对不上，JwtTokenProvider.validateToken 会捕获 JwtException 返回 false。除非攻击者拿到服务端的密钥，而密钥存在 application.yml 里，生产环境应该通过环境变量注入。"

### Q36: 你的项目能承受多少并发？

**标准回答**：
"这个项目是演示级别的，没做正式的压测。但架构上留了扩展空间：Spring Boot 默认 200 线程的 Tomcat 线程池；点击计数用了内存缓冲降低 DB 压力；热门数据可以走 Redis 缓存；前后端分离可以独立扩容。如果要支持更高并发，可以加 Nginx 负载均衡、数据库读写分离、消息队列削峰。"

### Q37: 如果要加新功能（比如标签系统），你会怎么设计？

**标准回答**：
"首先在数据库加 tags 表（tag_id, name）和 news_tags 关联表（news_id, tag_id）。后端加 TagController、TagService、TagMapper。前端在写稿页加标签选择器，在首页加标签筛选。这个项目已经有 tags 字段在 news 表里（逗号分隔），升级成关联表更规范。"

### Q38: 你的项目里有什么设计模式？

**标准回答**：
"用了单例模式（Spring 管理的 Bean 默认单例）、模板方法模式（ServiceImpl 提供基础 CRUD，我重写部分方法）、代理模式（AOP 切面是动态代理）、建造者模式（JWT 的 Jwts.builder()）。还用了 MVC、分层架构、依赖注入这些架构模式。"

## 十、最终必杀题

### Q39: 这个项目你最自豪的是什么？

**标准回答**：
"最自豪的是我从 PHP 原生版本一步步升级到 Spring Boot + Vue 3 的过程。不只是把代码搬过来，而是重新思考了架构设计——分层怎么分、安全怎么做、性能怎么优化、推荐算法怎么加。整个过程中我理解了'为什么 Spring Boot 这么设计'、'为什么 JWT 比 Session 更好'这些问题，而不只是会用。"

### Q40: 如果重新做，你会怎么改进？

**标准回答**：
"三个改进方向。第一，加上更完善的中文分词（现在用 Bigram 简单分词，可以用 HanLP 替换）。第二，引入消息队列（RabbitMQ/Kafka）处理异步任务——审核通过后的通知、ES 索引更新都用消息队列，提高系统解耦性。第三，加监控——Prometheus + Grafana 监控接口响应时间和错误率，ELK 收集日志。"

### Q41: 导师问你："你这不就是个 CRUD 吗？有什么技术含量？"

**杀手回答**：
"表面看是 CRUD，但里面有好几个不 CRUD 的设计。第一是内容推荐算法，TF-IDF + 余弦相似度，不是查数据库能解决的。第二是点击计数的 Write-Behind 优化，涉及并发编程和定时任务。第三是 JWT 双 token 的认证架构，涉及安全设计。第四是 Docker Compose 的 6 服务编排，涉及运维。另外审核工作流是状态机设计，不是简单的增删改查。如果只是 CRUD，我用 PHP 就做了，为什么要升级到 Spring Boot 呢？"
