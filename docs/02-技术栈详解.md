## 2.1 Maven —— 项目的"管家"

### 是什么
Maven 帮你做三件事：**管理依赖**、**编译打包**、**运行项目**。

### 类比
你要做一道菜，需要买菜（下载 jar 包）、切菜（编译）、炒菜（打包）、上桌（运行）。Maven 是你雇的帮厨，你只需要告诉他"做什么菜"（写 pom.xml），他帮你搞定一切。

### pom.xml 怎么读

```xml
<!-- 项目坐标：世界上唯一标识这个项目 -->
<groupId>com.example</groupId>       <!-- 组织 -->
<artifactId>news-server</artifactId>  <!-- 项目名 -->
<version>3.0.0</version>             <!-- 版本 -->

<!-- 父 POM：继承 Spring Boot 的默认配置 -->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.5</version>
</parent>

<!-- 依赖：项目需要哪些第三方库 -->
<dependencies>
    <!-- Spring Boot Web：内嵌 Tomcat + Spring MVC -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <!-- Spring Security：认证和授权 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-security</artifactId>
    </dependency>
    <!-- ... 更多依赖 -->
</dependencies>
```

### 常用命令

```bash
mvn compile         # 编译 Java 代码
mvn test            # 运行测试
mvn package         # 打包成 JAR
mvn spring-boot:run # 直接运行
mvn clean           # 删除 target 目录（清理编译产物）
```

### 面试可能问
- "Maven 和 Gradle 的区别？" → Maven 用 XML 配置，约定大于配置；Gradle 用 Groovy/Kotlin DSL，更灵活，构建更快
- "依赖传递是什么？" → A 依赖 B，B 依赖 C，A 自动获得 C。Maven 自动处理
- "依赖冲突怎么解决？" → 最短路径优先、先声明优先、用 `<exclusions>` 排除

## 2.2 Spring Boot —— 项目的"骨架"

### 是什么
Spring Boot 是一个**框架的框架**。它帮你整合了各种组件（Web、安全、数据库、缓存），你只需要关注业务代码。

### 核心概念

#### 2.2.1 控制反转（IoC）和依赖注入（DI）

**传统方式**（你自己 new 对象）：
```java
public class UserService {
    private UserMapper userMapper = new UserMapperImpl();  // 自己创建，紧耦合
}
```

**Spring 方式**（Spring 帮你 new 对象）：
```java
@Service
public class UserService {
    private final UserMapper userMapper;

    public UserService(UserMapper userMapper) {  // Spring 自动传入实现
        this.userMapper = userMapper;
    }
}
```

**好处**：你要换一个不同的 UserMapper 实现，改 Spring 配置就行，不用改业务代码。

#### 2.2.2 Bean（Spring 管理的对象）

任何被 Spring 创建和管理的对象都叫 Bean。怎么成为 Bean？
- 类上加 `@Service`、`@Component`、`@Controller`、`@Repository`
- `@Configuration` 类里的 `@Bean` 方法
- `@Mapper`（MyBatis-Plus 的注解）

#### 2.2.3 ApplicationContext（Spring 容器）

所有 Bean 都存在一个叫 ApplicationContext 的地方。你可以理解为一个巨大的 HashMap：
```
Map<String, Object> context = {
    "userService" → UserService 对象,
    "userMapper"  → UserMapper 对象,
    "passwordEncoder" → BCryptPasswordEncoder 对象,
    ...
}
```
Spring 在启动时扫描所有类，找到 `@Service`、`@Component` 等注解，创建对象，放进这个 Map。

#### 2.2.4 自动配置（Auto-Configuration）

Spring Boot 最强大的特性。当你引入 `spring-boot-starter-web`，它自动：
- 启动内嵌 Tomcat（Web 服务器，默认 8080 端口）
- 配置 Spring MVC（处理 HTTP 请求）
- 配置 Jackson（JSON 序列化）

你不需要写任何 XML 配置文件，一切自动完成。

### 项目中的体现

`NewsApplication.java`：
```java
@SpringBootApplication  // 这个注解包含了三个注解：
                        // @Configuration（可以定义 Bean）
                        // @EnableAutoConfiguration（自动配置）
                        // @ComponentScan（扫描当前包及子包的所有组件）
@MapperScan("com.example.news.modules.*.mapper")  // 扫描 MyBatis Mapper
@EnableAsync           // 开启异步（@Async 注解生效）
@EnableScheduling      // 开启定时任务（@Scheduled 注解生效）
public class NewsApplication {
    public static void main(String[] args) {
        SpringApplication.run(NewsApplication.class, args);
        // 这行代码启动了整个 Spring 容器，启动了 Tomcat，启动了所有组件
    }
}
```

### 面试可能问
- "Spring Boot 的自动配置原理？" → @SpringBootApplication → @EnableAutoConfiguration → 根据 classpath 中的 jar 自动创建 Bean
- "Bean 的生命周期？" → 实例化 → 属性赋值 → 初始化 → 使用 → 销毁
- "Bean 的作用域？" → singleton（默认，全局唯一）、prototype（每次 new）、request（每个请求）、session（每个会话）

## 2.3 Spring MVC —— 处理 HTTP 请求

### 是什么
Spring MVC 是 Spring 中专门处理 Web 请求的模块。MVC = Model（数据）+ View（页面）+ Controller（控制器）。

因为是前后端分离，我们用 `@RestController`（= @Controller + @ResponseBody，返回 JSON 而不是页面）。

### 一个 HTTP 请求的处理流程

```
浏览器发送 GET /api/v1/news/123
        │
        ▼
┌──────────────────┐
│ DispatcherServlet │  ← Spring 的前端控制器（所有请求的入口）
└──────┬───────────┘
       │ 根据 URL 路径找到对应的方法
       ▼
┌──────────────────┐
│ NewsController   │  ← @RestController
│ detail(123)      │  ← @GetMapping("/news/{id}") 匹配
└──────┬───────────┘
       │ 调用 newsService.getDetail(123)
       ▼
┌──────────────────┐
│ NewsService      │  ← @Service
│ getDetail(123)   │  ← 查数据库，组装数据
└──────┬───────────┘
       │ 返回 NewsDetailVO 对象
       ▼
┌──────────────────┐
│ Jackson          │  ← 自动把 Java 对象转成 JSON 字符串
└──────┬───────────┘
       │
       ▼
  HTTP 200
  {"code":200,"data":{"title":"xxx",...}}
```

### 常用注解

| 注解 | 作用 | 例子 |
|------|------|------|
| `@GetMapping("/news/{id}")` | 处理 GET 请求 | `GET /api/v1/news/123` |
| `@PostMapping("/news")` | 处理 POST 请求 | `POST /api/v1/news` |
| `@PutMapping("/news/{id}")` | 处理 PUT 请求 | `PUT /api/v1/news/123` |
| `@DeleteMapping("/news/{id}")` | 处理 DELETE 请求 | `DELETE /api/v1/news/123` |
| `@PathVariable` | 取 URL 路径中的变量 | `@PathVariable Long id` → 123 |
| `@RequestParam` | 取 URL 问号后面的参数 | `@RequestParam int page` → ?page=1 |
| `@RequestBody` | 取请求体中的 JSON | `@RequestBody NewsSaveDTO dto` |
| `@RequestHeader` | 取请求头 | `@RequestHeader("Authorization")` |

## 2.4 MyBatis-Plus —— 操作数据库

### 是什么
MyBatis-Plus 是 MyBatis 的增强版。让你用极少的代码操作数据库。

### 核心概念

#### Entity（实体类）
```java
@Data
@TableName("users")  // ← 指定对应的数据库表
public class User {
    @TableId(type = IdType.AUTO)  // ← 主键自增
    private Long userId;           // ← 自动映射到 user_id 列（驼峰转下划线）

    private String username;       // ← 自动映射到 username 列
    private String password;
    private String role;
    private Integer status;
}
```

#### Mapper（数据访问接口）
```java
@Mapper
public interface UserMapper extends BaseMapper<User> {
    // 继承 BaseMapper<User> 后，自动获得以下方法：
    // selectById(1L)         → SELECT * FROM users WHERE user_id=1
    // selectList(wrapper)    → SELECT * FROM users WHERE ...
    // insert(user)           → INSERT INTO users VALUES (...)
    // updateById(user)       → UPDATE users SET ... WHERE user_id=?
    // deleteById(1L)         → DELETE FROM users WHERE user_id=1
    // selectPage(page, wrapper) → 分页查询
    // selectCount(wrapper)   → SELECT COUNT(*) FROM users WHERE ...

    // 如果内置方法不够，可以自己写 SQL（本项目中没用到，全用内置方法）
}
```

#### LambdaQueryWrapper（条件构造器）
```java
// 查询：username='admin' 且 status=1 且 role='admin'
User user = userMapper.selectOne(new LambdaQueryWrapper<User>()
    .eq(User::getUsername, "admin")     // WHERE username = 'admin'
    .eq(User::getStatus, 1)             // AND status = 1
    .eq(User::getRole, "admin")         // AND role = 'admin'
);

// 模糊搜索
wrapper.like(User::getTitle, keyword);  // WHERE title LIKE '%keyword%'

// 排序
wrapper.orderByDesc(User::getCreatedAt); // ORDER BY created_at DESC

// BETWEEN 查询
wrapper.between(News::getCreatedAt, start, end);

// IN 查询
wrapper.in(News::getStatus, "published", "pending");

// 分组条件
wrapper.and(w -> w.like(News::getTitle, kw).or().like(News::getContent, kw));
// WHERE (title LIKE '%kw%' OR content LIKE '%kw%')
```

#### 分页
```java
// 查第 1 页，每页 10 条
Page<News> page = new Page<>(1, 10);
newsMapper.selectPage(page, wrapper);
// page.getRecords()  → 本页数据
// page.getTotal()    → 总条数
// page.getCurrent()  → 当前页码
```

#### ServiceImpl（服务层基类）
```java
@Service
public class NewsService extends ServiceImpl<NewsMapper, News> {
    // 继承后自动获得：
    // this.getById(id)       → 查一个
    // this.list(wrapper)     → 查列表
    // this.save(entity)      → 插入
    // this.updateById(entity) → 更新
    // this.removeById(id)    → 删除
    // this.page(page, wrapper) → 分页
    // this.count(wrapper)    → 计数

    // 可以重写这些方法，也可以添加自己的业务方法
}
```

### 面试可能问
- "MyBatis 和 JPA/Hibernate 的区别？" → MyBatis 写 SQL，灵活可控；JPA 自动生成 SQL，开发快但复杂查询难优化
- "MyBatis-Plus 比 MyBatis 多了什么？" → 自带 CRUD、分页、条件构造器、乐观锁、逻辑删除
- "#{} 和 ${} 的区别？" → #{} 是预编译（防 SQL 注入），${} 是字符串拼接（危险，只在表名/列名动态时用）

## 2.5 Spring Security —— 认证和授权

### 是什么
Spring Security 是一个安全框架，保护你的 API 不被未授权的人访问。

### 两个核心概念

**认证（Authentication）**：你是谁？
→ 用户输入用户名密码 → 验证通过 → 发 JWT token

**授权（Authorization）**：你能干什么？
→ 检查 token 中的角色 → admin 能删用户，user 不能

### 本项目的安全架构

```
HTTP 请求到达
    │
    ▼
┌─────────────────────┐
│ JwtAuthFilter       │  ← 最先执行
│ 1.从Header取token   │
│ 2.验证token是否有效 │
│ 3.解析出userId/role │
│ 4.设置到SecurityCtx│
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ SecurityFilterChain │  ← URL 权限检查
│ /auth/** → public   │
│ /admin/** → ADMIN   │
│ /audit/** → ADMIN   │
│   或 AUDITOR         │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ @PreAuthorize       │  ← 方法级权限检查
│ Controller 方法上的  │
│ 注解                 │
└──────┬──────────────┘
       │
       ▼
    业务逻辑
```

### SecurityConfig.java 详解

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity  // 开启方法级授权（@PreAuthorize）
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) {
        http
            // 1. 关闭 CSRF 保护（前后端分离 + JWT，不需要 CSRF）
            .csrf(AbstractHttpConfigurer::disable)

            // 2. 无状态 Session（JWT 自带信息，不需要服务端 Session）
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

            // 3. URL 级别的权限控制
            .authorizeHttpRequests(auth -> auth
                // 公开接口：登录、注册、API 文档、WebSocket
                .requestMatchers("/api/v1/auth/**").permitAll()
                .requestMatchers("/doc.html", "/swagger-ui/**").permitAll()
                // 公开接口：新闻列表和详情（可以不登录看）
                .requestMatchers(HttpMethod.GET, "/api/v1/news", "/api/v1/news/*").permitAll()
                .requestMatchers(HttpMethod.GET, "/api/v1/categories").permitAll()
                .requestMatchers(HttpMethod.GET, "/api/v1/recommend/**").permitAll()
                .requestMatchers(HttpMethod.GET, "/api/v1/slides").permitAll()

                // 需要登录
                .requestMatchers("/api/v1/user/**").authenticated()
                .requestMatchers("/api/v1/favorites/**").authenticated()
                .requestMatchers("/api/v1/likes/**").authenticated()
                .requestMatchers("/api/v1/reviews/**").authenticated()

                // 管理员专属
                .requestMatchers("/api/v1/admin/**").hasRole("ADMIN")
                // 管理员和审核员
                .requestMatchers("/api/v1/audit/**").hasAnyRole("ADMIN", "AUDITOR")
                // 管理员和编辑
                .requestMatchers("/api/v1/editor/**").hasAnyRole("ADMIN", "EDITOR")

                // 其他所有请求需要登录
                .anyRequest().authenticated()
            )

            // 4. 在用户名密码认证过滤器之前，插入我们的 JWT 过滤器
            .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

### 面试可能问
- "Session 和 JWT 的区别？" → Session 在服务端存状态，JWT 在客户端存状态。JWT 无状态便于水平扩展
- "CSRF 是什么？为什么这里关了？" → 跨站请求伪造，前后端分离 + JWT 天然免疫
- "Spring Security 的过滤器链？" → 能讲清楚上面那个流程图就行

## 2.6 JWT（JSON Web Token）—— 项目中的实现

### Token 的完整生命周期

```
1. 用户登录 POST /api/v1/auth/login { username, password }
                     │
2. 服务器验证用户名密码                              │
                     │
3. 生成 Access Token（30min）                     │
   生成 Refresh Token（7天，存 Redis）              │
                     │
4. 返回两个 Token 给前端                            │
                     │
5. 前端存 localStorage                             │
   每次请求带 Authorization: Bearer <accessToken>   │
                     │
6. 30分钟后 Access Token 过期                      │
                     │
7. 前端自动用 Refresh Token 换新的 Access Token      │
   POST /api/v1/auth/refresh { refreshToken }      │
                     │
8. 7天后 Refresh Token 也过期 → 重新登录            │
```

### JWT Token 的结构

JWT 是三段 Base64 编码的字符串，用 `.` 分隔：

```
eyJhbGciOiJIUzM4NCJ9          ← Header：算法类型（HMAC-SHA384）
.
eyJyb2xlIjoiYWRtaW4iLC...     ← Payload：实际数据（角色、用户ID、过期时间）
.
KkfJUc3wrczrT_tDb-mhj4...     ← Signature：签名（防篡改）
```

Payload 解码后：
```json
{
  "role": "admin",
  "userId": 1,
  "username": "admin",
  "sub": "1",
  "iat": 1783063790,    // 签发时间
  "exp": 1783065590     // 过期时间（30分钟后）
}
```

### JwtTokenProvider.java 详解

```java
@Component
public class JwtTokenProvider {

    private final SecretKey secretKey;  // 签名密钥（从配置文件读取）
    private final long accessTokenExpiration;   // 30分钟
    private final long refreshTokenExpiration;  // 7天

    // 构造函数：从 application.yml 读取 jwt.secret 和过期时间
    // 用 HMAC-SHA 算法创建密钥
    public JwtTokenProvider(@Value("${jwt.secret}") String secret, ...) {
        this.secretKey = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }

    // 生成 Access Token
    public String generateAccessToken(Long userId, String username, String role) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", userId);
        claims.put("username", username);
        claims.put("role", role);           // ← 角色信息存在 token 里

        return Jwts.builder()
            .claims(claims)
            .subject(String.valueOf(userId))
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + accessTokenExpiration))
            .signWith(secretKey)            // ← 用密钥签名
            .compact();
    }

    // 解析 Token
    public Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(secretKey)          // ← 用同样的密钥验证
            .build()
            .parseSignedClaims(token)       // 如果签名不对或过期，抛异常
            .getPayload();
    }

    // 验证 Token 是否有效
    public boolean validateToken(String token) {
        try {
            parseToken(token);
            return true;
        } catch (ExpiredJwtException e) {
            // 过期了 → false
        } catch (JwtException e) {
            // 签名不对或其他问题 → false
        }
        return false;
    }
}
```

### JwtAuthenticationFilter.java 详解

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    // OncePerRequestFilter：保证每个请求只执行一次

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain) {
        // 1. 从请求头获取 token
        String token = extractToken(request);  // "Bearer xxx" → "xxx"

        if (token != null) {
            // 2. 检查是否在黑名单（登出后的 token 不能用）
            if (redisUtil != null && redisUtil.hasKey("blacklist:" + token)) {
                chain.doFilter(request, response);  // 不放行也不设认证
                return;
            }

            // 3. 验证 token 有效性
            if (jwtTokenProvider.validateToken(token)) {
                // 4. 解析出用户信息
                String username = jwtTokenProvider.getUsername(token);
                String role = jwtTokenProvider.getRole(token);
                Long userId = jwtTokenProvider.getUserId(token);

                // 5. 创建认证对象并设置到 Spring Security 上下文
                UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(username, userId, authorities);
                SecurityContextHolder.getContext().setAuthentication(authentication);
                // ↑ 之后的所有代码都能通过 SecurityContextHolder 获取当前用户
            }
        }

        // 6. 放行，继续执行下一个过滤器
        chain.doFilter(request, response);
    }
}
```

### Refresh Token 轮转机制（RefreshTokenService.java）

```java
@Service
@ConditionalOnBean(RedisUtil.class)  // Redis 不可用时自动禁用
public class RefreshTokenService {

    // 创建 Refresh Token：生成随机 UUID，存 Redis（7天过期）
    public String createRefreshToken(Long userId) {
        String token = UUID.randomUUID().toString().replace("-", "");
        redisUtil.setString("refresh_token:" + token, String.valueOf(userId), 7, TimeUnit.DAYS);
        return token;
    }

    // 验证 Refresh Token：查 Redis 看是否存在
    public Long validateRefreshToken(String token) {
        String userIdStr = redisUtil.getString("refresh_token:" + token);
        return userIdStr != null ? Long.parseLong(userIdStr) : null;
    }

    // 撤销：验证后立即删除旧的，实现"一次一换"
    public void revokeRefreshToken(String token) {
        redisUtil.delete("refresh_token:" + token);
    }

    // 黑名单：登出后的 Access Token 加入黑名单（TTL = 剩余有效时间）
    public void blacklistAccessToken(String accessToken, long ttlSeconds) {
        redisUtil.setString("blacklist:" + accessToken, "1", ttlSeconds, TimeUnit.SECONDS);
    }
}
```

### 面试可能问
- "JWT 怎么防止被篡改？" → 第三部分的签名是用密钥计算的，篡改后签名不匹配
- "JWT 的缺点？" → token 一旦签发无法撤销（用短有效期 + 黑名单缓解）、payload 不加密（不要存敏感信息）
- "为什么用双 token？" → access 短有效期保证安全，refresh 长有效期保证体验
- "Refresh Token Rotation 是什么？" → 每次使用 refresh token 换 access token 时，旧的 refresh token 立即失效，换一个新的。就算被盗，原主再次使用时会发现 token 已失效

## 2.7 BCrypt —— 密码加密

### 为什么不用 MD5
```
MD5("123456") = e10adc3949ba59abbe56e057f20f883e
MD5("123456") = e10adc3949ba59abbe56e057f20f883e  ← 每次都一样！
```
- 相同的密码产生相同的哈希 → 查彩虹表就能反向破解
- 计算速度极快 → 跑字典几秒钟试几十万个密码

```
BCrypt("123456") = $2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
BCrypt("123456") = $2a$10$TzNg0hVLMGJClDqUOboFfOJqLJxVQwGnHw8EzMqJK5WqDpEvhK7qy
// 同样的密码，每次都不同！（因为随机盐值）
```
- 自带随机盐值（salt），彩虹表攻击无效
- 计算速度故意慢（cost factor=10，约 100ms），暴力破解极慢
- 验证用 `matches("123456", hash)` → true，不需要知道盐值

### 项目中的使用

```java
// PasswordConfig.java - 创建 BCrypt 编码器
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
    // cost factor=10，即 2^10 = 1024 轮哈希
}

// UserService.java - 注册时加密
user.setPassword(passwordEncoder.encode("123456"));
// 存到数据库：$2a$10$...

// 登录时验证
if (!passwordEncoder.matches(inputPassword, user.getPassword())) {
    throw new BusinessException(ErrorCode.PASSWORD_ERROR);
}
```

### 面试可能问
- "BCrypt 和 MD5 的区别？" → 上面那个表
- "盐值是什么？" → 随机字符串，拼在密码前面一起哈希，让相同的密码产生不同的结果
- "cost factor 是什么？" → 计算轮数，越大越慢（也越安全），2^cost 次哈希

## 2.8 AOP（面向切面编程）—— 项目中的日志和限流

### 是什么
在不修改原有代码的情况下，在方法执行前后插入额外的逻辑。

### 类比
你去餐厅吃饭，点菜（调用方法）。AOP = 餐厅门口的服务员：
- 进门时问"有预订吗？"（限流检查）
- 你点菜时记下来（记录日志）
- 你吃完后帮你结账（记录操作）

厨师只管做菜（业务逻辑），服务员管接待（AOP 横切关注点）。

### 项目中的两个 AOP

**LogAspect.java —— 操作日志**
```java
@Aspect   // 声明这是一个切面
@Component
public class LogAspect {

    @Around("@annotation(logOperation)")  // 拦截所有带 @LogOperation 的方法
    public Object around(ProceedingJoinPoint joinPoint, LogOperation logOp) {
        // 方法执行前
        Object result = joinPoint.proceed();  // 执行原方法
        // 方法执行后：异步记录日志（存数据库）
        saveLog(logOp.value());
        return result;
    }
}

// 使用：
@LogOperation(value = "登录系统")
public Result login(...) { ... }
// → 调用 login 时，自动记录一条日志："xxx 登录系统"
```

**RateLimitAspect.java —— 接口限流**
```java
@Aspect
@Component
@ConditionalOnBean(StringRedisTemplate.class)  // 有 Redis 才生效
public class RateLimitAspect {

    @Around("@annotation(rateLimit)")
    public Object around(ProceedingJoinPoint joinPoint, RateLimit rateLimit) {
        String key = "rate_limit:" + rateLimit.key() + ":" + userId;
        Long count = redisTemplate.opsForValue().increment(key);  // +1
        if (count == 1) redisTemplate.expire(key, rateLimit.window(), SECONDS);

        if (count > rateLimit.limit()) {
            throw new BusinessException("操作太频繁，请稍后重试");
        }

        return joinPoint.proceed();  // 放行
    }
}

// 使用：
@RateLimit(key = "login", limit = 10, window = 60)
public Result login(...) { ... }
// → 同一用户 60 秒内最多登录 10 次
```

### 面试可能问
- "AOP 的原理？" → 动态代理：JDK 代理（接口）或 CGLIB 代理（类），在运行时生成代理对象
- "AOP 的应用场景？" → 日志、事务管理、权限校验、性能监控、限流

## 2.9 Redis —— 缓存和限流

### 是什么
一个基于内存的 key-value 数据库，数据存在内存里，速度极快（微秒级）。

### 为什么快
- 数据在内存（RAM），不是硬盘
- 单线程处理命令，无锁竞争
- 简单数据结构，优化极好

### 在本项目中的三个用途

**用途 1：数据缓存**
```
第一次查询热门新闻 → 查 MySQL（慢，100ms）→ 结果放 Redis（快，1ms）
第二次查询热门新闻 → 直接从 Redis 取（1ms）
30分钟后缓存过期 → 重新查 MySQL
```

**用途 2：点击计数缓冲（Write-Behind 模式）**
```
每次阅读 → Redis INCR news:clicks:{id}  （内存操作，1ms）
每 5 分钟 → 读取所有 click 值，批量 UPDATE MySQL
```
这样做的好处：1000 次阅读 = 1000 次 Redis INCR（内存，快） + 1 次 MySQL UPDATE（批量），而不是 1000 次 MySQL UPDATE（每次都锁行）。

**用途 3：限流计数器**
```
同一用户 60 秒内请求 login 接口：
第 1 次 → Redis INCR rate:login:user123 → 1，放行
第 2 次 → Redis INCR → 2，放行
...
第 11 次 → Redis INCR → 11 > 10，拒绝！
```

### Redis 数据结构

| 类型 | 场景 | 命令 |
|------|------|------|
| String | 缓存、计数器、分布式锁 | SET/GET/INCR/SETNX |
| Hash | 存储对象（用户信息） | HSET/HGET/HGETALL |
| List | 消息队列、最新列表 | LPUSH/RPOP |
| Set | 标签、去重 | SADD/SMEMBERS |
| ZSet | 排行榜 | ZADD/ZRANGE |

项目主要用 String（缓存、token）和 ZSet（排行榜）。

### 面试可能问
- "缓存穿透、击穿、雪崩分别是什么？怎么解决？" → 详见第 09 篇
- "Redis 持久化？" → RDB（快照）和 AOF（追加日志）
- "Redis 为什么单线程还这么快？" → 内存操作 + 非阻塞 I/O 多路复用

## 2.10 Elasticsearch —— 全文搜索

### 是什么
专业的搜索引擎。不是数据库，专门用来做搜索。

### 和 MySQL LIKE 的本质区别

MySQL SELECT * FROM news WHERE title LIKE '%人工智能%'：
```
扫描 news 表的每一行 → 检查 title 是否包含"人工智能" → 返回匹配的行
100 万条数据 → 大约 2 秒
```

Elasticsearch 搜索"人工智能"：
```
查倒排索引 → 直接定位到包含"人工智能"的文档 ID → 返回
100 万条数据 → 大约 10 毫秒
```

### 倒排索引是什么

正排索引（书前面的目录）：
```
文档1 → [人工, 智能, 辅助, 诊断]
文档2 → [数字, 人民币, 试点]
```

倒排索引（书后面的索引）：
```
人工 → [文档1, 文档5, 文档12]
智能 → [文档1, 文档3, 文档8]
辅助 → [文档1, 文档9]
```
搜索"人工智能" → 取"人工"和"智能"的文档列表交集 → [文档1]

### 中文分词（ik 分词器）
```
"人工智能辅助诊断系统"
→ ik_max_word（细粒度）: [人工智能, 人工, 智能, 辅助, 诊断, 系统]
→ ik_smart（粗粒度）:   [人工智能, 辅助, 诊断, 系统]
```

### 项目中的策略
- Elasticsearch 可用时：用 ES 搜索，结果含高亮
- Elasticsearch 不可用时：自动降级为 MySQL LIKE 搜索
- 这个降级策略体现了系统的**高可用性**设计

### 面试可能问
- "ES 和 MySQL 搜索的区别？" → 上面那张表
- "倒排索引的原理？" → 词 → 文档列表的映射
- "ES 怎么保证数据一致性？" → 近实时（NRT），默认 1 秒刷新

## 2.12 Docker —— 容器化部署

### 是什么
把应用和它需要的所有环境打包成一个"容器"，保证在任何机器上运行结果一致。

### 关键概念

**镜像（Image）**：一个只读模板，包含运行应用所需的一切（代码、运行时、库、环境变量）

**容器（Container）**：镜像的运行实例

**Dockerfile**：定义如何构建镜像

**Docker Compose**：定义和运行多个容器

### 本项目的 Dockerfile（后端）

```dockerfile
# 第一阶段：构建（用 Maven 编译打包）
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B    # 下载依赖（缓存）
COPY src ./src
RUN mvn clean package -DskipTests   # 编译打包

# 第二阶段：运行（只复制 JAR，不包含 Maven 源码）
FROM eclipse-temurin:17-jre-alpine  # 只需 JRE，更小
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

为什么两阶段？最终镜像不包含 Maven 和源码，体积从 500MB 减到 150MB。

### docker-compose.yml

```yaml
services:
  mysql:         # MySQL 8.0 数据库
  redis:         # Redis 7 缓存
  elasticsearch: # ES 8 搜索
  minio:         # MinIO 文件存储
  server:        # Spring Boot 后端（依赖 mysql 和 redis 先启动）
  web:           # Vue + Nginx 前端（依赖 server）
```

一条命令启动全部：`docker compose up -d`

### 面试可能问
- "Docker 和虚拟机的区别？" → Docker 共享宿主机内核，更轻量；虚拟机有完整 OS
- "多阶段构建有什么好处？" → 减小镜像体积，不包含构建工具和源码
