2 第 1 篇：Java 基础补充——看懂项目代码的最低门槛

你已掌握：变量、循环、if-else、方法定义、类的创建。但是看项目代码时会遇到泛型、注解、Lambda、Stream 这些没学过的东西。这篇专门补这五个知识点。

2.1 1. 泛型（Generics）

2.1.1 是什么

把类型当作参数传递。List<String> 意思是”这个列表里只能放字符串”。

2.1.2 为什么需要

没有泛型时：

List list = new ArrayList();
list.add("hello");
list.add(123);        // 也能加进去！编译器不报错
String s = (String) list.get(0);  // 必须强转，写错了运行时才报错

有泛型后：

List<String> list = new ArrayList<>();
list.add("hello");
// list.add(123);     // 编译就报错！IDE 会标红
String s = list.get(0);  // 不需要强转

2.1.3 项目中哪里用到了

几乎到处都是。举几个例子：

// Result.java - 统一返回结果
public class Result<T> {        // <T> 表示"任意类型"
    private T data;             // data 的类型由调用者决定

    public static <T> Result<T> ok(T data) {
        return new Result<>(200, "success", data);
    }
}

// 使用时：
Result<UserVO> r1 = Result.ok(userVO);      // T = UserVO
Result<String> r2 = Result.ok("hello");      // T = String
Result<List<NewsVO>> r3 = Result.ok(list);   // T = List<NewsVO>
// BaseMapper.java - MyBatis-Plus 的泛型接口
public interface UserMapper extends BaseMapper<User> {
    // BaseMapper<User> 的意思是"这个 Mapper 操作的是 User 表"
    // 继承了 selectById、insert、updateById 等方法，泛型参数自动确定为 User
}

2.1.4 面试可能问

  • “泛型的作用是什么？” → 编译期类型检查，避免运行时 ClassCastException
  • “泛型擦除是什么？” → Java 编译后泛型信息被擦除，运行时 List 和 List 是同一个类

2.2 2. 注解（Annotation）

2.2.1 是什么

给代码贴标签。@Override 告诉编译器”这个方法是重写父类的”。

2.2.2 项目中常见的注解

Spring 框架注解（最重要）：

注解 位置 作用 大白话
@SpringBootApplication NewsApplication.java 标记启动类 “我是程序入口”
@RestController Controller 类上 标记为 REST 接口类 “这个类对外暴露 API”
@Service Service 类上 标记为业务逻辑类 “这个类处理业务”
@Component 工具类上 标记为 Spring 管理的 Bean “Spring 帮我创建和管理这个对象”
@Autowired 字段上 自动注入依赖 “Spring 帮我把这个字段赋值”
@Bean @Configuration 类的方法上 手动创建 Bean “用这个方法创建的对象交给 Spring 管理”
@Value("${...}") 字段上 读取配置文件 “从 application.yml 取这个值”

数据相关注解：

注解 作用
@Data (Lombok) 自动生成 getter/setter/toString/equals/hashCode
@TableName("users") (MyBatis-Plus) 指定实体类对应的数据库表名
@TableId(type = IdType.AUTO) 主键自增
@TableField(fill = FieldFill.INSERT) 插入时自动填充（如创建时间）

校验注解：

注解 作用
@NotBlank(message = "用户名不能为空") 字符串不能为空或全空格
@Size(min = 2, max = 50) 字符串长度限制
@Email 必须是合法邮箱格式
@Valid 触发校验

安全注解：

注解 作用
@PreAuthorize("hasRole('ADMIN')") 只有管理员能调用这个方法
@PreAuthorize("hasAnyRole('ADMIN','AUDITOR')") 管理员或审核员能调用

自定义注解（本项目的）：

注解 作用
@LogOperation(value = "登录系统") 标记这个方法需要记录操作日志
@RateLimit(key = "login", limit = 10, window = 60) 限制这个方法 60 秒内最多调用 10 次

2.2.3 注解怎么工作的？

以 @LogOperation 为例： 1. 定义注解 @interface LogOperation { String value(); } 2. 写一个 AOP 切面类 LogAspect，用 @Around("@annotation(logOperation)") 拦截所有带这个注解的方法 3. 方法执行时，AOP 在方法执行前后插入日志记录逻辑

// 1. 定义
public @interface LogOperation {
    String value() default "";
}

// 2. 使用
@LogOperation(value = "登录系统")
public Result login(...) { ... }

// 3. AOP 拦截
@Around("@annotation(logOperation)")
public Object around(ProceedingJoinPoint jp, LogOperation op) {
    // 方法执行前
    Object result = jp.proceed();  // 执行原方法
    // 方法执行后：记录日志
    saveLog(op.value());
    return result;
}

2.3 3. 反射（Reflection）

2.3.1 是什么

在运行时动态获取类的信息、调用方法、修改字段。正常代码是”编译时就知道要调什么”，反射是”运行时才决定调什么”。

2.3.2 项目中哪里用到了

  • Spring 的 @Autowired：Spring 扫描到你的字段，用反射给它赋值
  • MyBatis-Plus 的 selectById(1L)：框架用反射把数据库查出来的 ResultSet 变成 User 对象
  • AOP：运行时动态创建代理对象，在方法前后插入逻辑

2.3.3 面试可能问

  • “反射的优缺点？” → 灵活但性能较低，破坏封装性
  • “反射怎么用？” → Class.forName()、getDeclaredMethod()、method.invoke()

2.4 4. Lambda 表达式

2.4.1 是什么

一种简洁的写法，把方法当参数传递。

// 传统写法：匿名内部类
new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println("hello");
    }
}).start();

// Lambda 写法：一样的功能，代码少很多
new Thread(() -> System.out.println("hello")).start();

2.4.2 项目中哪里用到了

// LambdaQueryWrapper 的条件构造
userMapper.selectOne(new LambdaQueryWrapper<User>()
    .eq(User::getUsername, "admin")       // User::getUsername 是方法引用
    .eq(User::getStatus, 1)
);

// Stream 操作
List<NewsVO> voList = newsList.stream()
    .map(n -> toVO(n))                   // n -> toVO(n) 是 Lambda
    .filter(vo -> vo != null)            // 过滤空值
    .collect(Collectors.toList());       // 收集成 List

2.5 5. Stream API

2.5.1 是什么

一种函数式操作集合的方式，链式调用，像流水线一样处理数据。

2.5.2 核心操作

List<String> names = users.stream()
    .filter(u -> u.getStatus() == 1)     // 过滤：只要启用的用户
    .map(u -> u.getUsername())           // 转换：User → String
    .sorted()                            // 排序
    .limit(10)                           // 取前 10 个
    .collect(Collectors.toList());       // 收集成 List

// 统计
long count = users.stream().count();
boolean allActive = users.stream().allMatch(u -> u.getStatus() == 1);

// 转 Map
Map<Long, String> userMap = users.stream()
    .collect(Collectors.toMap(User::getUserId, User::getUsername));

2.5.3 项目中哪里用到了

NewsService.java 中大量使用 Stream 做数据转换：

Map<Long, String> userMap = userMapper.selectList(null).stream()
    .collect(Collectors.toMap(User::getUserId, User::getUsername));
// 把 List<User> 转成 Map<userId, username>，后面 O(1) 查找

2.6 6. Optional

2.6.1 是什么

一个容器，可能包含值也可能为空，强迫你处理空值情况。

2.6.2 项目中哪里用到了

// ServletUtils.java
public static Optional<HttpServletRequest> getCurrentRequest() {
    ServletRequestAttributes attrs = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
    return Optional.ofNullable(attrs).map(ServletRequestAttributes::getRequest);
}
// 调用方必须处理可能为空的情况

2.7 7. 访问修饰符

修饰符 本类 同包 子类 任何地方
private ✅ ❌ ❌ ❌
default(不写) ✅ ✅ ❌ ❌
protected ✅ ✅ ✅ ❌
public ✅ ✅ ✅ ✅

面试时回答”封装”相关问题用：private 字段 + public getter/setter = 控制访问权限。

2.8 8. static vs 实例

static 非 static（实例）
属于谁 类本身 类的每个对象
怎么调用 类名.方法() 对象.方法()
内存 只有一份 每个对象各一份 
public class Result<T> {
    // 静态方法：不需要 new Result() 就能用
    public static <T> Result<T> ok(T data) { ... }

    // 实例方法：需要先 new Result() 才能用
    public int getCode() { return this.code; }
}

// 使用
Result<User> r = Result.ok(user);   // 静态方法直接用类名调
int code = r.getCode();             // 实例方法用对象调

2.9 9. final 关键字

用法 含义
final class X 这个类不能被继承
final void method() 这个方法不能被子类重写
final int x = 5 这个变量只能赋值一次（常量）
final User user = ... 引用不能变（不能指向别的对象），但对象内部可以变

项目中大量使用 private final 声明依赖：

private final UserService userService;  // 构造函数赋值后不可变

2.10 10. this 关键字

  • this.name → 当前对象的 name 字段（区别于参数 name）
  • this() → 调用本类的另一个构造方法
  • this.getById() → 调用自己的方法
  • 在构造方法中：this.name = name;（区分字段和参数）

3.1 2.1 Maven —— 项目的”管家”

3.1.1 是什么

Maven 帮你做三件事：管理依赖、编译打包、运行项目。

3.1.2 类比

你要做一道菜，需要买菜（下载 jar 包）、切菜（编译）、炒菜（打包）、上桌（运行）。Maven 是你雇的帮厨，你只需要告诉他”做什么菜”（写 pom.xml），他帮你搞定一切。

3.1.3 pom.xml 怎么读

<!-- 项目坐标：世界上唯一标识这个项目 -->
<groupId>com.example</groupId>       <!-- 组织 -->
<artifactId>news-server</artifactId>  <!-- 项目名 -->
<version>3.0.0</version>             <!-- 版本 -->

<!-- 父 POM：继承 Spring Boot 的默认配置 -->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.5</version>
</parent>

<!-- 依赖：项目需要哪些第三方库 -->
<dependencies>
    <!-- Spring Boot Web：内嵌 Tomcat + Spring MVC -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <!-- Spring Security：认证和授权 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-security</artifactId>
    </dependency>
    <!-- ... 更多依赖 -->
</dependencies>

3.1.4 常用命令

mvn compile         # 编译 Java 代码
mvn test            # 运行测试
mvn package         # 打包成 JAR
mvn spring-boot:run # 直接运行
mvn clean           # 删除 target 目录（清理编译产物）

3.1.5 面试可能问

  • “Maven 和 Gradle 的区别？” → Maven 用 XML 配置，约定大于配置；Gradle 用 Groovy/Kotlin DSL，更灵活，构建更快
  • “依赖传递是什么？” → A 依赖 B，B 依赖 C，A 自动获得 C。Maven 自动处理
  • “依赖冲突怎么解决？” → 最短路径优先、先声明优先、用 <exclusions> 排除

3.2 2.2 Spring Boot —— 项目的”骨架”

3.2.1 是什么

Spring Boot 是一个框架的框架。它帮你整合了各种组件（Web、安全、数据库、缓存），你只需要关注业务代码。

3.2.2 核心概念

3.2.2.1 2.2.1 控制反转（IoC）和依赖注入（DI）

传统方式（你自己 new 对象）：

public class UserService {
    private UserMapper userMapper = new UserMapperImpl();  // 自己创建，紧耦合
}

Spring 方式（Spring 帮你 new 对象）：

@Service
public class UserService {
    private final UserMapper userMapper;

    public UserService(UserMapper userMapper) {  // Spring 自动传入实现
        this.userMapper = userMapper;
    }
}

好处：你要换一个不同的 UserMapper 实现，改 Spring 配置就行，不用改业务代码。

3.2.2.2 2.2.2 Bean（Spring 管理的对象）

任何被 Spring 创建和管理的对象都叫 Bean。怎么成为 Bean？ - 类上加 @Service、@Component、@Controller、@Repository - @Configuration 类里的 @Bean 方法 - @Mapper（MyBatis-Plus 的注解）

@EnableMethodSecurity  // 开启方法级授权（@PreAuthorize）

3.6 2.6 JWT（JSON Web Token）—— 项目中的实现

3.6.1 Token 的完整生命周期

1. 用户登录 POST /api/v1/auth/login { username, password }
                     │
2. 服务器验证用户名密码                              │
                     │
3. 生成 Access Token（30min）                     │
   生成 Refresh Token（7天，存 Redis）              │
                     │
4. 返回两个 Token 给前端                            │
                     │
5. 前端存 localStorage                             │
   每次请求带 Authorization: Bearer <accessToken>   │
                     │
6. 30分钟后 Access Token 过期                      │
                     │
7. 前端自动用 Refresh Token 换新的 Access Token      │
   POST /api/v1/auth/refresh { refreshToken }      │
                     │
8. 7天后 Refresh Token 也过期 → 重新登录            │

3.6.2 JWT Token 的结构

JWT 是三段 Base64 编码的字符串，用 . 分隔：

eyJhbGciOiJIUzM4NCJ9          ← Header：算法类型（HMAC-SHA384）
.
eyJyb2xlIjoiYWRtaW4iLC...     ← Payload：实际数据（角色、用户ID、过期时间）
.
KkfJUc3wrczrT_tDb-mhj4...     ← Signature：签名（防篡改）

Payload 解码后：

{
  "role": "admin",
  "userId": 1,
  "username": "admin",
  "sub": "1",
  "iat": 1783063790,    // 签发时间
  "exp": 1783065590     // 过期时间（30分钟后）
}

3.6.3 JwtTokenProvider.java 详解

@Component
public class JwtTokenProvider {

    private final SecretKey secretKey;  // 签名密钥（从配置文件读取）
    private final long accessTokenExpiration;   // 30分钟
    private final long refreshTokenExpiration;  // 7天

    // 构造函数：从 application.yml 读取 jwt.secret 和过期时间
    // 用 HMAC-SHA 算法创建密钥
    public JwtTokenProvider(@Value("${jwt.secret}") String secret, ...) {
        this.secretKey = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }

    // 生成 Access Token
    public String generateAccessToken(Long userId, String username, String role) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", userId);
        claims.put("username", username);
        claims.put("role", role);           // ← 角色信息存在 token 里

        return Jwts.builder()
            .claims(claims)
            .subject(String.valueOf(userId))
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + accessTokenExpiration))
            .signWith(secretKey)            // ← 用密钥签名
            .compact();
    }

    // 解析 Token
    public Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(secretKey)          // ← 用同样的密钥验证
            .build()
            .parseSignedClaims(token)       // 如果签名不对或过期，抛异常
            .getPayload();
    }

    // 验证 Token 是否有效
    public boolean validateToken(String token) {
        try {
            parseToken(token);
            return true;
        } catch (ExpiredJwtException e) {
            // 过期了 → false
        } catch (JwtException e) {
            // 签名不对或其他问题 → false
        }
        return false;
    }
}

3.6.4 JwtAuthenticationFilter.java 详解

@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    // OncePerRequestFilter：保证每个请求只执行一次

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain) {
        // 1. 从请求头获取 token
        String token = extractToken(request);  // "Bearer xxx" → "xxx"

        if (token != null) {
            // 2. 检查是否在黑名单（登出后的 token 不能用）
            if (redisUtil != null && redisUtil.hasKey("blacklist:" + token)) {
                chain.doFilter(request, response);  // 不放行也不设认证
                return;
            }

            // 3. 验证 token 有效性
            if (jwtTokenProvider.validateToken(token)) {
                // 4. 解析出用户信息
                String username = jwtTokenProvider.getUsername(token);
                String role = jwtTokenProvider.getRole(token);
                Long userId = jwtTokenProvider.getUserId(token);

                // 5. 创建认证对象并设置到 Spring Security 上下文
                UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(username, userId, authorities);
                SecurityContextHolder.getContext().setAuthentication(authentication);
                // ↑ 之后的所有代码都能通过 SecurityContextHolder 获取当前用户
            }
        }

        // 6. 放行，继续执行下一个过滤器
        chain.doFilter(request, response);
    }
}

3.6.5 Refresh Token 轮转机制（RefreshTokenService.java）

@Service
@ConditionalOnBean(RedisUtil.class)  // Redis 不可用时自动禁用
public class RefreshTokenService {

    // 创建 Refresh Token：生成随机 UUID，存 Redis（7天过期）
    public String createRefreshToken(Long userId) {
        String token = UUID.randomUUID().toString().replace("-", "");
        redisUtil.setString("refresh_token:" + token, String.valueOf(userId), 7, TimeUnit.DAYS);
        return token;
    }

    // 验证 Refresh Token：查 Redis 看是否存在
    public Long validateRefreshToken(String token) {
        String userIdStr = redisUtil.getString("refresh_token:" + token);
        return userIdStr != null ? Long.parseLong(userIdStr) : null;
    }

    // 撤销：验证后立即删除旧的，实现"一次一换"
    public void revokeRefreshToken(String token) {
        redisUtil.delete("refresh_token:" + token);
    }

    // 黑名单：登出后的 Access Token 加入黑名单（TTL = 剩余有效时间）
    public void blacklistAccessToken(String accessToken, long ttlSeconds) {
        redisUtil.setString("blacklist:" + accessToken, "1", ttlSeconds, TimeUnit.SECONDS);
    }
}

3.6.6 面试可能问

  • “JWT 怎么防止被篡改？” → 第三部分的签名是用密钥计算的，篡改后签名不匹配
  • “JWT 的缺点？” → token 一旦签发无法撤销（用短有效期 + 黑名单缓解）、payload 不加密（不要存敏感信息）
  • “为什么用双 token？” → access 短有效期保证安全，refresh 长有效期保证体验
  • “Refresh Token Rotation 是什么？” → 每次使用 refresh token 换 access token 时，旧的 refresh token 立即失效，换一个新的。就算被盗，原主再次使用时会发现 token 已失效

3.7 2.7 BCrypt —— 密码加密

3.7.1 为什么不用 MD5

MD5("123456") = e10adc3949ba59abbe56e057f20f883e
MD5("123456") = e10adc3949ba59abbe56e057f20f883e  ← 每次都一样！
  • 相同的密码产生相同的哈希 → 查彩虹表就能反向破解
  • 计算速度极快 → 跑字典几秒钟试几十万个密码
BCrypt("123456") = $2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
BCrypt("123456") = $2a$10$TzNg0hVLMGJClDqUOboFfOJqLJxVQwGnHw8EzMqJK5WqDpEvhK7qy
// 同样的密码，每次都不同！（因为随机盐值）
  • 自带随机盐值（salt），彩虹表攻击无效
  • 计算速度故意慢（cost factor=10，约 100ms），暴力破解极慢
  • 验证用 matches("123456", hash) → true，不需要知道盐值

3.7.2 项目中的使用

// PasswordConfig.java - 创建 BCrypt 编码器
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
    // cost factor=10，即 2^10 = 1024 轮哈希
}

// UserService.java - 注册时加密
user.setPassword(passwordEncoder.encode("123456"));
// 存到数据库：$2a$10$...

// 登录时验证
if (!passwordEncoder.matches(inputPassword, user.getPassword())) {
    throw new BusinessException(ErrorCode.PASSWORD_ERROR);
}

3.7.3 面试可能问

  • “BCrypt 和 MD5 的区别？” → 上面那个表
  • “盐值是什么？” → 随机字符串，拼在密码前面一起哈希，让相同的密码产生不同的结果
  • “cost factor 是什么？” → 计算轮数，越大越慢（也越安全），2^cost 次哈希

@Aspect
@Component
@ConditionalOnBean(StringRedisTemplate.class)  // 有 Redis 才生效
public class RateLimitAspect {

    @Around("@annotation(rateLimit)")
    public Object around(ProceedingJoinPoint joinPoint, RateLimit rateLimit) {
        String key = "rate_limit:" + rateLimit.key() + ":" + userId;
        Long count = redisTemplate.opsForValue().increment(key);  // +1
        if (count == 1) redisTemplate.expire(key, rateLimit.window(), SECONDS);

        if (count > rateLimit.limit()) {
            throw new BusinessException("操作太频繁，请稍后重试");
        }

        return joinPoint.proceed();  // 放行
    }
}

// 使用：
@RateLimit(key = "login", limit = 10, window = 60)
public Result login(...) { ... }
// → 同一用户 60 秒内最多登录 10 次

3.9 2.9 Redis —— 缓存和限流

3.9.1 是什么

一个基于内存的 key-value 数据库，数据存在内存里，速度极快（微秒级）。

3.9.2 为什么快

  • 数据在内存（RAM），不是硬盘
  • 单线程处理命令，无锁竞争
  • 简单数据结构，优化极好

3.9.3 在本项目中的三个用途

用途 1：数据缓存

第一次查询热门新闻 → 查 MySQL（慢，100ms）→ 结果放 Redis（快，1ms）
第二次查询热门新闻 → 直接从 Redis 取（1ms）
30分钟后缓存过期 → 重新查 MySQL

用途 2：点击计数缓冲（Write-Behind 模式）

每次阅读 → Redis INCR news:clicks:{id}  （内存操作，1ms）
每 5 分钟 → 读取所有 click 值，批量 UPDATE MySQL

这样做的好处：1000 次阅读 = 1000 次 Redis INCR（内存，快） + 1 次 MySQL UPDATE（批量），而不是 1000 次 MySQL UPDATE（每次都锁行）。

用途 3：限流计数器

同一用户 60 秒内请求 login 接口：
第 1 次 → Redis INCR rate:login:user123 → 1，放行
第 2 次 → Redis INCR → 2，放行
...
第 11 次 → Redis INCR → 11 > 10，拒绝！

3.9.4 Redis 数据结构

类型 场景 命令
String 缓存、计数器、分布式锁 SET/GET/INCR/SETNX
Hash 存储对象（用户信息） HSET/HGET/HGETALL
List 消息队列、最新列表 LPUSH/RPOP
Set 标签、去重 SADD/SMEMBERS
ZSet 排行榜 ZADD/ZRANGE

项目主要用 String（缓存、token）和 ZSet（排行榜）。

3.10.1 是什么

4.2.1.1 UserService.java —— 最核心的业务逻辑

@Service
public class UserService extends ServiceImpl<UserMapper, User> {

    // 登录验证
    public User login(LoginDTO dto) {
        // 1. 查用户（只按用户名查，不再需要选角色）
        User user = this.getOne(new LambdaQueryWrapper<User>()
            .eq(User::getUsername, dto.getUsername()));

        // 2. 验证密码（BCrypt.matches 而不是 ==）
        if (user == null || !passwordEncoder.matches(dto.getPassword(), user.getPassword())) {
            throw new BusinessException(ErrorCode.PASSWORD_ERROR);
        }

        // 3. 检查状态
        if (user.getStatus() == 0) {
            throw new BusinessException(ErrorCode.ACCOUNT_DISABLED);
        }

        return user;
    }

    // 注册（重要安全点！）
    public void register(RegisterDTO dto) {
        // 1. 检查用户名唯一
        if (this.count(new LambdaQueryWrapper<User>()
                .eq(User::getUsername, dto.getUsername())) > 0) {
            throw new BusinessException(ErrorCode.USERNAME_EXISTS);
        }

        // 2. 检查邮箱唯一（如果提供了）
        if (dto.getEmail() != null && !dto.getEmail().isEmpty()) {
            if (this.count(new LambdaQueryWrapper<User>()
                    .eq(User::getEmail, dto.getEmail())) > 0) {
                throw new BusinessException(1007, "该邮箱已被注册");
            }
        }

        // 3. 创建用户
        User user = new User();
        user.setUsername(dto.getUsername());
        user.setPassword(passwordEncoder.encode(dto.getPassword()));  // BCrypt 加密！
        user.setRole("user");   // ← 强制普通用户！不接受前端传来的角色
        user.setEmail(dto.getEmail());
        user.setPhone(dto.getPhone());
        user.setStatus(1);      // 默认启用
        this.save(user);
    }
}

面试关键点： - user.setRole("user") 强制角色，这是安全设计！防止有人注册时传 role=admin - 登录不再需要选角色（删除了 .eq(User::getRole, dto.getRole())），系统自动识别

5.2 二、认证与安全（高频）

5.2.1 Q7: JWT 和 Session 有什么区别？为什么选 JWT？

Session JWT
状态存储 服务端（内存/Redis） 客户端（token 本身）
扩展性 需要共享 Session（粘性会话或 Redis） 天然无状态，任意服务器都能验证
注销 直接删 Session 需要黑名单机制
适用场景 单体应用 分布式/微服务

“我选 JWT 因为前后端分离 + 无状态方便扩展。但 JWT 有个缺点——签发了就无法撤销。所以用短有效期（30分钟）+ 黑名单机制解决。”

5.2.2 Q8: 双 token 怎么设计的？为什么这样设计？

标准回答： “Access Token 有效期 30 分钟，每次请求都带。Refresh Token 有效期 7 天，存在 Redis 里，Access Token 过期后用 Refresh Token 换新的。这样设计的好处：Access Token 短 → 即使被盗 30 分钟后就失效；Refresh Token 长 → 不用频繁登录。还有一个细节：Refresh Token 是’一次一换’的，每次使用后旧的立即失效，换一个新的，防止重放攻击。”

追问：“如果 Refresh Token 也被盗了呢？” → “这种情况说明客户端设备已经被完全控制。但我们的 Refresh Token 一次一换机制可以检测到：如果攻击者和原主同时使用同一个 Refresh Token，其中一个人会失败，系统可以发通知提醒用户。”

5.2.3 Q9: BCrypt 比 MD5 好在哪里？

标准回答： “MD5 有三个致命缺陷：相同的密码产生相同的哈希 → 彩虹表能批量破解；计算速度极快 → 暴力破解几秒钟试几十万次；没有盐值。BCrypt 解决了这三个问题：自动生成随机盐值，相同密码每次结果不同；计算速度故意慢（cost factor=10，约 100ms），暴力破解成本极高。现在是业界标准，Spring Security 默认推荐。”

5.2.4 Q10: 你的注册接口安全吗？怎么防止恶意注册？

标准回答： “有三层防护。第一，注册强制为普通用户角色，即使有人伪造请求传 role=admin，后端也强制设置为 user。第二，加了 @RateLimit 限流，同一用户 60 秒内最多注册 5 次。第三，密码长度 6-50 位限制。生产环境还可以加邮箱验证码和图形验证码，因为需要 SMTP 服务器所以开发环境没做。”

5.2.5 Q11: Spring Security 的过滤器链是什么？

标准回答： “过滤器链是一串按顺序执行的过滤器，每个过滤器做一件事。我的项目里，JwtAuthenticationFilter 在 UsernamePasswordAuthenticationFilter 之前执行。它从请求头提取 token，验证有效性，解析出用户信息，设到 SecurityContext 里。然后 SecurityFilterChain 根据 URL 规则检查权限。”

5.4 四、推荐算法（加分关键）

5.4.1 Q17: 你的推荐系统怎么做的？

标准回答： “用的是基于内容的推荐。步骤如下：第一步，用 Unigram+Bigram 对中文文本做简单分词；第二步，对每个词计算 TF-IDF 值形成文档向量；第三步，用余弦相似度计算当前文章和其他文章的相似度；第四步，取 Top5 返回。另外有 LRU 缓存避免重复计算，冷启动时降级到同分类推荐。”

5.4.2 Q18: TF-IDF 是什么？怎么算的？

标准回答： “TF 是词频，一个词在文档中出现次数除以文档总词数，衡量这个词对本文有多重要。IDF 是逆文档频率，总文档数除以包含该词的文档数的对数，衡量这个词的稀有度。TF 乘 IDF 就是最终的权重。一个词在本文出现多但在其他文档出现少 → TF-IDF 值高 → 对本文很重要。”

5.4.3 Q19: 余弦相似度是什么？

标准回答： “余弦相似度衡量两个向量方向的一致性。公式是 A·B / (|A|×|B|)。两个文档向量夹角越小（余弦值越接近 1），内容越相似。比如一篇讲 AI 的文章和一篇讲机器学习的文章，它们的 TF-IDF 向量方向接近，余弦相似度高。”

5.4.4 Q20: 冷启动问题怎么解决？

标准回答： “基于内容的推荐天然冷启动友好，因为不需要用户行为数据，只要有文章内容就行。如果文章内容太短（少于 100 字），TF-IDF 效果不好，就降级到同分类高点击推荐。后续用户行为数据积累多了，可以加上协同过滤。”

追问：“协同过滤和内容推荐有什么区别？” → “内容推荐基于文章本身的特征（标题、正文），不管用户行为。协同过滤基于’相似用户喜欢相似内容’的假设，需要大量用户行为数据。我的项目两者结合：内容推荐做兜底，行为数据多了就上协同过滤。”

5.5 五、缓存与性能（深度话题）

5.5.1 Q21: 点击计数为什么不直接 UPDATE？

标准回答： “直接 UPDATE 有两个问题。第一是行锁竞争：高并发时多个请求同时 UPDATE 同一行，MySQL 加行锁，后来的请求排队等。第二是写放大：每次阅读都触发一次磁盘写入。我的方案是 Write-Behind 模式：先用 ConcurrentHashMap 在内存累加计数，每 5 分钟通过 @Scheduled 定时任务批量 UPDATE 一次，把 1000 次写变成 1 次写。”

5.5.2 Q22: 缓存穿透、击穿、雪崩分别是什么？

问题 定义 本项目的解法
穿透 查不存在的数据，每次穿透到 DB 空值也缓存（短 TTL）
击穿 热点数据过期瞬间，大量请求打 DB 分布式锁（SETNX），拿到锁的去查 DB，其他的等
雪崩 大量缓存同时过期，DB 瞬间崩溃 随机 TTL（在基础 TTL 上加随机偏移）

5.5.3 Q23: 分布式锁怎么实现的？

标准回答： “用 Redis 的 SETNX 命令。SET key value NX EX timeout，NX 表示只在 key 不存在时设置（互斥），EX 是过期时间（防死锁）。拿到锁的线程去查数据库重建缓存，没拿到的等 100ms 再重试或直接返回旧缓存。”

6.1 话题 1：缓存三大问题——穿透、击穿、雪崩

6.1.1 缓存穿透

是什么：查一个根本不存在的数据。比如查 newsId=99999，数据库里没有，缓存也没有。每次请求都穿透缓存打到数据库。

怎么解决：

// 方案1：缓存空值
News news = cache.get(key);
if (news == null) {
    news = db.query(key);
    if (news == null) {
        cache.set(key, EMPTY_OBJECT, 60);  // 空值也缓存 60 秒
        return null;
    }
    cache.set(key, news, 1800);  // 有值缓存 30 分钟
}
return news;

// 方案2：布隆过滤器
// 把所有存在的 newsId 预先放入布隆过滤器
// 请求来时先问布隆："newsId=99999 在不在？"→"不在"→直接返回
// 布隆说"在"→查缓存→查 DB

面试怎么讲：“缓存穿透的典型解法是缓存空值和布隆过滤器。对于 ID 不存在这种场景，缓存空值最直接。对于更复杂的查询条件，可以用布隆过滤器预判。”

6.1.2 缓存击穿

是什么：热点数据缓存刚好过期，瞬间大量请求同时打到数据库。比如一篇 10 万阅读的热门新闻，缓存刚好过期，1000 个请求同时查。

怎么解决：

public News getHotNews(Long newsId) {
    News news = cache.get(key);
    if (news != null) return news;

    // 分布式锁：只有一个请求能拿到锁去查 DB
    String lockKey = "lock:news:" + newsId;
    boolean locked = redis.setIfAbsent(lockKey, "1", 10, SECONDS);

    if (locked) {
        try {
            news = db.query(newsId);     // 查 DB
            cache.set(key, news, 1800);  // 重建缓存
        } finally {
            redis.delete(lockKey);       // 释放锁
        }
    } else {
        Thread.sleep(100);               // 没拿到锁，等 100ms
        return cache.get(key);           // 重试读缓存（可能已经有值了）
    }
    return news;
}

面试怎么讲：“击穿的解决思路是’互斥’——同一时刻只让一个请求去查数据库重建缓存。我用 Redis 的 SETNX 实现分布式锁，拿到锁的去查 DB，没拿到的等一会再读缓存。”

6.1.3 缓存雪崩

是什么：大量缓存同时过期。比如所有新闻缓存都是 30 分钟 TTL，同时创建的缓存同时过期，瞬间所有请求都打到数据库。

怎么解决：

// 随机 TTL：在基础 TTL 上加 0~5 分钟的随机偏移
int ttl = 1800 + ThreadLocalRandom.current().nextInt(300);
cache.set(key, news, ttl);

面试怎么讲：“雪崩的核心是’不要同时过期’。在基础 TTL 上加随机偏移，让过期时间分散，避免同一时刻大量缓存失效。另外可以做缓存预热——启动时预加载热点数据进缓存。”

6.2 话题 2：JWT 深入——无状态认证的权衡

6.2.1 为什么 JWT 令牌签发了就无法撤销？

JWT 是无状态的，服务端不存任何信息。用户拿着 token 来，服务端验证签名通过就认为是合法用户。不像 Session——服务端可以删掉 Session 让用户立即登出。

6.2.2 本项目的解决方案

方案 1：短有效期（治本）
Access Token 只有 30 分钟。即使签发了无法撤销，
30 分钟后 token 自动失效。风险窗口只有 30 分钟。

方案 2：黑名单（治标）
用户主动登出时，把 token 加入 Redis 黑名单，
TTL = token 剩余有效时间。下次请求时检查黑名单。

方案 3：Refresh Token Rotation（检测）
Refresh Token 一次一换。如果同一个 token 被用了两次，
说明有人盗用，系统可以标记异常并通知用户。

6.2.3 面试怎么讲

“JWT 的无状态性是一把双刃剑。好处是扩展性强，坏处是无法撤销。我用了三个机制缓解：短有效期降低风险窗口、黑名单支持主动登出、Refresh Token Rotation 检测盗用。实际生产环境通常还会配合 OAuth2.0 的授权码模式。”

6.3 话题 3：Write-Behind vs Write-Through

6.3.1 两种缓存更新策略

Write-Through（写穿）：写缓存的同时写数据库。强一致性，但每次都写磁盘。

Write-Behind（写后）：只写缓存，异步批量写数据库。性能好，但可能丢数据。

6.3.2 本项目的选择

点击计数用 Write-Behind：

// 写操作：只写内存
clickBuffer.merge(newsId, 1L, Long::sum);  // 微秒级

// 刷库：定时批量
@Scheduled(fixedRate = 300000)
public void flushClickCounts() {
    // 批量 UPDATE，1000 次变 1 次
}

为什么？ 点击计数对一致性的要求不高——少算几个点击量不影响业务。但性能要求高——热门文章每秒可能有几十次点击。

6.3.3 可能丢数据吗？怎么解决？

@PreDestroy
public void flushOnShutdown() {
    flushClickCounts();  // JVM 关闭前强制刷一次
}

如果 JVM 崩溃（kill -9），最近 5 分钟的点击数据会丢失。对于新闻点击量这种场景可以接受。如果是交易金额，就不能用 Write-Behind。

6.3.4 面试怎么讲

“点击计数是典型的’高性能、低一致性要求’场景，所以选了 Write-Behind。我加了 @PreDestroy 钩子减少正常关机的数据丢失，也承认异常崩溃会丢少量数据——但这不是交易系统，可以接受。”

6.4 话题 4：推荐算法的数学原理

6.4.1 TF-IDF 公式

TF(t, d) = 词 t 在文档 d 中出现的次数 / 文档 d 的总词数

IDF(t) = log( (总文档数 + 1) / (包含词 t 的文档数 + 1) ) + 1

TF-IDF(t, d) = TF(t, d) × IDF(t)

6.4.2 为什么 IDF 要加 1 平滑？

避免 IDF 为 0（当词在所有文档中都出现时），也避免除零错误。

6.4.3 余弦相似度公式

cos(θ) = (A · B) / (|A| × |B|)

A · B = Σ(Ai × Bi)       点积：两个向量对应元素相乘再求和
|A| = √(Σ Ai²)           模长：向量各元素的平方和再开方

6.4.4 数值实例

假设有两篇文章： - 文档A：“人工智能改变世界” - 文档B：“人工智能辅助医疗诊断”

分词后： - 文档A：{人工:1, 智能:1, 改变:1, 世界:1} - 文档B：{人工:1, 智能:1, 辅助:1, 医疗:1, 诊断:1}

余弦相似度计算：

共同词"人工"和"智能"贡献点积
|A| = √(1+1+1+1) = 2
|B| = √(1+1+1+1+1) = √5 ≈ 2.236
cos(θ) = (1×1 + 1×1) / (2 × 2.236) = 2 / 4.472 ≈ 0.447

约 0.45 的相似度，不算很高但有一定相关性。相比之下，一篇讲”体育比赛”的文章相似度可能只有 0.05。

6.4.5 面试怎么讲

“TF-IDF 的核心思想是：一个词在一篇文章中出现得多但在其他文章中很少，那这个词就是这篇文章的关键词。用这些关键词形成向量，再用余弦相似度算两篇文章的夹角——夹角越小越相似。”

6.5 话题 5：并发编程——ConcurrentHashMap 为什么线程安全

6.5.1 HashMap 为什么线程不安全？

HashMap 在扩容（resize）时会重新分配桶（bucket）。多线程同时 put 时，可能导致链表成环（死循环），或者数据丢失。

6.5.2 ConcurrentHashMap 怎么解决？

Java 8 的 ConcurrentHashMap 用 CAS（Compare-And-Swap）+ synchronized 实现： - 读操作不加锁（volatile 保证可见性） - 写操作只锁当前桶的头节点，不影响其他桶 - 扩容时多线程协作迁移数据

6.5.3 本项目中的使用

private final ConcurrentHashMap<Long, Long> clickBuffer = new ConcurrentHashMap<>();

public void incrementClick(Long newsId) {
    clickBuffer.merge(newsId, 1L, Long::sum);
    // merge 是原子操作，读-改-写一气呵成
    // 如果用 HashMap + put + get，在高并发下会丢数据
}

6.5.4 面试怎么讲

“ConcurrentHashMap 比 Hashtable 更高效，因为不是锁整个表而是锁分段。Java 8 后进一步优化为 CAS + 桶级锁，读操作完全无锁。我用它的 merge 方法做原子累加，替代了传统的 get-put 两步操作，避免了竞态条件。”

6.6 话题 6：Spring 的循环依赖

6.6.1 是什么

A 依赖 B，B 依赖 A。Spring 创建 A 时需要 B，创建 B 时需要 A → 死循环。

@Service
class A {
    @Autowired B b;
}
@Service
class B {
    @Autowired A a;
}

6.6.2 Spring 怎么解决

三级缓存： 1. singletonObjects（成品池）：完全创建好的 Bean 2. earlySingletonObjects（半成品池）：创建中但属性未填充的 Bean 3. singletonFactories（工厂池）：能生产半成品的工厂

流程： 1. 创建 A → 发现需要 B → 把 A 的半成品放入三级缓存 → 去创建 B 2. 创建 B → 发现需要 A → 从三级缓存取 A 的半成品 → 注入给 B → B 创建完成 3. 回到 A → 此时 B 已完成 → 注入 B → A 创建完成

6.6.3 本项目中

本项目的依赖关系是一棵树（Controller → Service → Mapper），不存在循环依赖。但如果面试被问到，能讲清楚三级缓存机制。

6.7 话题 7：SQL 注入和你项目中的防护

6.7.1 SQL 注入是什么

// 危险代码！
String username = request.getParameter("username");  // 用户输入: ' OR '1'='1
String sql = "SELECT * FROM users WHERE username='" + username + "'";
// 实际执行: SELECT * FROM users WHERE username='' OR '1'='1'
// → 返回所有用户！

6.7.2 本项目的防护

MyBatis-Plus 的 LambdaQueryWrapper 使用预编译语句（PreparedStatement）：

userMapper.selectOne(new LambdaQueryWrapper<User>()
    .eq(User::getUsername, username)
);
// 生成的 SQL: SELECT * FROM users WHERE username = ?
// 参数 username 作为 ? 的值传入，永远不会被当作 SQL 执行

所有 SQL 关键字（列名、表名）都在编译时确定，用户输入的只能是值。

6.7.3 面试怎么讲

“LambdaQueryWrapper 本质是参数化查询，用户输入作为 PreparedStatement 的参数绑定，不会被当作 SQL 执行。另外 MyBatis 的 #{} 也是预编译，但 ${} 是字符串拼接——项目中只用 #{} 或 LambdaQueryWrapper，杜绝了注入风险。”

6.8 话题 8：RESTful API 设计原则

6.8.1 本项目的 API 设计

GET    /api/v1/news          → 列表
GET    /api/v1/news/123      → 详情
POST   /api/v1/news          → 创建
PUT    /api/v1/news/123      → 更新

7.3.2 路由守卫 (beforeEach) 逐行解析

router.beforeEach(async (to, from, next) => {
  // to：要去的页面   from：来自哪个页面   next：放行/跳转

  const userStore = useUserStore()

  // 第 1 步：公开页面直接放行
  if (to.meta.public) return next()

  // 第 2 步：未登录 → 强制去登录页
  if (!userStore.isLoggedIn) return next('/login')

  // 第 3 步：已登录但还没加载用户信息 → 先加载
  if (!userStore.username) {
    try {
      await userStore.fetchUserInfo()  // 调 /auth/me 接口
    } catch {
      userStore.logout()               // token 过期 → 登出
      return next('/login')
    }
  }

  // 第 4 步：已登录却访问登录页 → 按角色跳转
  if (to.path === '/login') {
    return next(userStore.isUser ? '/' : '/admin')
  }

  // 第 5 步：角色不匹配 → 403
  if (to.meta.roles && !to.meta.roles.includes(userStore.role)) {
    return next('/403')
  }

  // 第 6 步：一切正常 → 放行
  next()
})

这段代码保证了：未登录的人永远看不到需要登录的页面，角色不对的人永远看不到不该看的页面。

7.4 四、状态管理 —— stores/user.js

import { defineStore } from 'pinia'

export const useUserStore = defineStore('user', () => {
  // ===== 状态（数据） =====
  const accessToken = ref('')      // JWT Access Token
  const refreshToken = ref('')     // JWT Refresh Token
  const userId = ref(null)         // 用户 ID
  const username = ref('')         // 用户名
  const role = ref('')             // 角色：admin/auditor/editor/user
  const roleName = ref('')         // 角色中文名：管理员/审核员/编辑/普通用户
  const avatar = ref('')           // 头像（预留）

  // ===== 计算属性 =====
  const isLoggedIn = computed(() => !!accessToken.value)  // 有 token = 已登录
  const isAdmin = computed(() => role.value === 'admin')
  const isAuditor = computed(() => role.value === 'auditor')
  const isEditor = computed(() => role.value === 'editor')
  const isUser = computed(() => role.value === 'user')

  // ===== 方法 =====
  async function login(credentials) {
    const data = await request.post('/auth/login', credentials)
    accessToken.value = data.accessToken
    refreshToken.value = data.refreshToken
    userId.value = data.userId
    username.value = data.username
    role.value = data.role
    roleName.value = data.roleName
    // 此时 Pinia 持久化插件自动把这些写入 localStorage
  }

  async function logout() {
    // 1. 通知后端登出（把 token 加入黑名单）
    if (refreshToken.value) {
      request.post('/auth/logout', { refreshToken: refreshToken.value }).catch(() => {})
    }
    // 2. 清空本地状态
    accessToken.value = ''
    refreshToken.value = ''
    userId.value = null
    username.value = ''
    role.value = ''
    roleName.value = ''
    // localStorage 也会被自动清空（因为持久化监听 ref 变化）
  }

  async function fetchUserInfo() {
    const data = await request.get('/auth/me')
    userId.value = data.userId
    username.value = data.username
    role.value = data.role
    roleName.value = data.roleName
    avatar.value = data.avatar
  }

  return {
    accessToken, refreshToken, userId, username, role, roleName, avatar,
    isLoggedIn, isAdmin, isAuditor, isEditor, isUser,
    login, register, fetchUserInfo, logout
  }
}, {
  // 持久化配置：这些字段存 localStorage，刷新页面不丢
  persist: {
    key: 'news-user',
    pick: ['accessToken', 'refreshToken', 'userId', 'username', 'role', 'roleName', 'avatar']
    // 注意：password 不在列表里，永远不会存本地！
  }
})

import axios from 'axios'

const request = axios.create({
  baseURL: '/api/v1',     // 所有请求自动加这个前缀
  timeout: 15000          // 15 秒超时
})

// ===== 请求拦截器：发请求前做的事 =====
request.interceptors.request.use(
  config => {
    const userStore = useUserStore()
    if (userStore.accessToken) {
      // 自动带 Authorization 头
      config.headers.Authorization = `Bearer ${userStore.accessToken}`
    }
    return config
  },
  error => Promise.reject(error)
)

// ===== 响应拦截器：收到响应后做的事 =====
request.interceptors.response.use(
  response => {
    const { data } = response
    if (data.code === 200) {
      return data.data  // ← 自动解包！调用方直接拿到 data 里的内容
    }
    // 业务错误：弹提示
    ElMessage.error(data.message || '请求失败')
    return Promise.reject(new Error(data.message))
  },
  async error => {
    // HTTP 错误
    const { config, response } = error

    // Token 过期 → 自动刷新
    if (response?.status === 401 && !config._retry) {
      const userStore = useUserStore()
      if (!userStore.refreshToken) {
        userStore.logout()
        router.push('/login')
        return Promise.reject(error)
      }

      // 防止多个请求同时刷新 token（用队列）
      if (isRefreshing) {
        return new Promise((resolve) => {
          refreshQueue.push(resolve)
        }).then(() => {
          config.headers.Authorization = `Bearer ${userStore.accessToken}`
          return request(config)  // 重试原请求
        })
      }

      config._retry = true
      isRefreshing = true

      try {
        const { accessToken, refreshToken } = await refreshTokenApi()
        userStore.accessToken = accessToken
        userStore.refreshToken = refreshToken
        // 通知队列中等待的请求
        refreshQueue.forEach(resolve => resolve())
        refreshQueue = []
        // 重试原请求
        config.headers.Authorization = `Bearer ${accessToken}`
        return request(config)
      } catch (e) {
        refreshQueue.forEach(resolve => resolve()) // 即使失败也释放队列
        refreshQueue = []
        userStore.logout()
        router.push('/login')
        return Promise.reject(e)
      } finally {
        isRefreshing = false
      }
    }

    ElMessage.error(response?.data?.message || '网络错误')
    return Promise.reject(error)
  }
)

关键设计：token 刷新队列 如果有 3 个请求同时因为 token 过期返回 401，只有一个会真正去刷新 token，另外 2 个排队等待。刷新成功后，3 个请求都用新 token 重试。

7.8 八、前端面试要点

7.8.1 为什么用 Vue 3 Composition API？

比 Options API 更灵活：逻辑可以抽成 composables 复用，不需要 this，TypeScript 支持更好。

7.8.2 ref 和 reactive 的区别？

  • ref：包装基本类型（字符串、数字），需要 .value 访问
  • reactive：包装对象，直接访问属性，不需要 .value
  • 我的习惯：单个值用 ref，表单对象用 reactive

7.8.3 v-if 和 v-show 的区别？

  • v-if：条件为假时 DOM 元素不存在（销毁和重建），切换开销大
  • v-show：条件为假时 DOM 还在，只是 display:none，切换开销小
  • 审核队列用 v-show 切换全文：频繁切换，用 v-show 性能更好

7.8.4 computed 和 method 的区别？

  • computed：有缓存，依赖不变不重新计算
  • method：每次调用都重新执行
  • 示例：isAdmin 用 computed，因为 role 不变就不用重新判断

7.8.5 组件间怎么通信？

  • 父子：props（父→子）+ emit（子→父）
  • 跨组件：Pinia store（全局状态）
  • 路由参数：route.params.id（页面间传参）

8.2 二、每张表详解

8.2.1 2.1 users —— 用户表

CREATE TABLE users (
    user_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    username   VARCHAR(50) NOT NULL UNIQUE,    -- 用户名，唯一
    password   VARCHAR(255) NOT NULL,           -- BCrypt 哈希（v2: VARCHAR(32) MD5）
    email      VARCHAR(100) DEFAULT NULL,       -- v3 新增
    phone      VARCHAR(20) DEFAULT NULL,        -- v3 新增
    role       ENUM('admin','auditor','editor','user') NOT NULL DEFAULT 'editor',
    avatar     VARCHAR(255) DEFAULT NULL,       -- 头像路径
    status     TINYINT(1) DEFAULT 1,            -- 1=启用 0=停用
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

字段设计理由：

设计决策 理由
password VARCHAR(255) BCrypt 哈希固定 60 字符，255 留足余量（v2 的 32 只够 MD5）
role 用 ENUM 角色固定 4 种，ENUM 比 VARCHAR 省空间、有约束、查询快
status TINYINT 布尔值用 TINYINT 不用 BOOLEAN（MySQL BOOLEAN 本质是 TINYINT）
email 可 NULL 注册时选填，不强制
created_at DEFAULT CURRENT_TIMESTAMP 自动记录注册时间，不用代码设置

v2 → v3 变更： - password 从 VARCHAR(32) 扩到 VARCHAR(255)（MD5 → BCrypt） - 新增 email、phone 字段 - 字符集从 utf8 升级到 utf8mb4（支持 emoji）

8.2.2 2.2 categories —— 新闻分类表

CREATE TABLE categories (
    category_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(30) NOT NULL,
    sort_order  INT DEFAULT 0,          -- 排序（决定首页导航显示顺序）
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

种子数据：体育(1)、财经(2)、科技(3)、文化(4)、时政(5)、教育(6)、民生(7)

sort_order 的作用：首页导航和侧边栏按这个字段排序，管理员可以调整顺序。

8.2.3 2.3 news —— 新闻表（核心）

CREATE TABLE news (
    news_id       BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id       BIGINT NOT NULL,              -- 作者
    category_id   BIGINT NOT NULL,              -- 所属分类
    title         VARCHAR(100) NOT NULL,        -- 标题
    summary       VARCHAR(255) DEFAULT NULL,    -- 摘要（v3 新增）
    content       TEXT,                         -- 正文
    tags          VARCHAR(255) DEFAULT NULL,    -- 标签（v3 新增，逗号分隔）
    source        VARCHAR(100) DEFAULT NULL,    -- 来源（v3 新增）
    cover_image   VARCHAR(255) DEFAULT NULL,    -- 封面图路径
    status        ENUM('draft','pending','published','rejected') DEFAULT 'draft',
    reject_reason VARCHAR(255) DEFAULT NULL,    -- 退回原因
    is_top        TINYINT(1) DEFAULT 0,         -- 是否置顶（v3：置顶=轮播）
    clicked       BIGINT DEFAULT 0,             -- 点击量
    is_es_synced  TINYINT(1) DEFAULT 0,         -- ES 同步标记（v3 新增）
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (category_id) REFERENCES categories(category_id),

    INDEX idx_status_cat_time (status, category_id, created_at),  -- v3 新增
    INDEX idx_user_status (user_id, status)                        -- v3 新增
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

status 状态机：

draft ──→ pending ──→ published
  ↑                    (审核通过)
  │         └──→ rejected （审核退回）
  └──────────────┘（修改后重新提交）

索引设计理由：

索引 覆盖查询 为什么
idx_status_cat_time WHERE status='published' AND category_id=1 ORDER BY created_at DESC 首页分类筛选是最频繁的查询，联合索引三个字段覆盖完整查询
idx_user_status WHERE user_id=3 AND status='pending' “我的稿件”按状态筛选，编辑最常用的查询

v2 → v3 变更： - 新增 summary、tags、source 字段 - clicked 从 INT 改为 BIGINT（支持过亿点击） - 新增两个联合索引 - is_top 在 v3 中有新含义：置顶 = 出现在首页轮播

8.2.4 2.4 reviews —— 评论表（嵌套回复）

CREATE TABLE reviews (
    review_id  BIGINT AUTO_INCREMENT PRIMARY KEY,
    news_id    BIGINT NOT NULL,              -- 属于哪篇新闻
    user_id    BIGINT DEFAULT NULL,          -- 评论者（NULL=匿名）
    parent_id  BIGINT DEFAULT NULL,          -- 父评论ID（NULL=顶级评论）
    username   VARCHAR(50) DEFAULT '匿名',   -- 显示用的用户名
    content    TEXT NOT NULL,
    state      ENUM('pending','approved') DEFAULT 'approved',
    ip         VARCHAR(45) DEFAULT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE,
    FOREIGN KEY (parent_id) REFERENCES reviews(review_id) ON DELETE CASCADE,

    INDEX idx_news_state (news_id, state, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

嵌套回复怎么实现的？

顶级评论A (parent_id=NULL)
  ├── 回复A1 (parent_id=A.review_id)
  └── 回复A2 (parent_id=A.review_id)
顶级评论B (parent_id=NULL)
  └── 回复B1 (parent_id=B.review_id)

查询时先查 parent_id IS NULL 的顶级评论，再查 parent_id IN (...) 的子回复。后端 ReviewController 在内存中组装成嵌套结构。

为什么用 parent_id 而不是另建表？ - 一张表实现无限层级嵌套 - 不用额外的关联表 - 查询简单：两次 SELECT 即可

ON DELETE CASCADE：新闻被删除时，它的所有评论自动删除。父评论被删除时，子回复也删除。

为什么 IP 是 VARCHAR(45)？ IPv4 最长 15 字符（如 192.168.1.1），IPv6 最长 45 字符。

CREATE TABLE logs (

8.2.7 2.7 favorites —— 收藏表

CREATE TABLE favorites (
    fav_id     BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    news_id    BIGINT NOT NULL,

为什么单独建表而不是在 users 表加字段？ 收藏是多对多关系（一个用户收藏多篇新闻，一篇新闻被多人收藏）。关系必然要单独建表。

8.2.11 2.11 user_behavior —— 用户行为表（v3 新增）

CREATE TABLE user_behavior (
    behavior_id    BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id        BIGINT NOT NULL,
    news_id        BIGINT NOT NULL,
    behavior_type  ENUM('view','like','favorite','comment') NOT NULL,
    behavior_score DECIMAL(3,2) DEFAULT 1.00,  -- 行为权重：view(0.3), like(0.7), favorite(1.0), comment(0.5)
    created_at     DATETIME DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE,
    FOREIGN KEY (news_id) REFERENCES news(news_id) ON DELETE CASCADE,
    INDEX idx_user_behavior (user_id, behavior_type, created_at),
    INDEX idx_news_behavior (news_id, behavior_type)

为什么需要这张表？ 这是协同过滤推荐的数据源。不同行为有不同的权重： - 浏览(view)：0.3 分（看过不一定喜欢） - 点赞(like)：0.7 分（喜欢才会点） - 收藏(favorite)：1.0 分（非常喜欢才收藏） - 评论(comment)：0.5 分（可能是负面评论）

目前的状态：表已建好，代码还没写入（推荐系统目前用的是基于内容的 TF-IDF）。面试时可以讲”数据模型已建好，后续可以升级为混合推荐”。

8.3 三、为什么用 InnoDB 而不是 MyISAM？

InnoDB MyISAM
事务 ✅ 支持 ❌ 不支持
外键 ✅ 支持 ❌ 不支持
行级锁 ✅ ❌（表级锁）
崩溃恢复 ✅ ❌
适用场景 需要数据完整性的业务系统 只读/日志归档

本项目有审核流、点赞收藏等写操作，必须用 InnoDB。

8.4 四、为什么用 utf8mb4 而不是 utf8？

MySQL 的 utf8 实际上是阉割版，每个字符最多 3 字节，不支持 emoji（4 字节）。utf8mb4 是完整版，支持所有 Unicode 字符。

8.5 五、外键的取舍

本项目中用了外键，但实际生产环境大厂经常不用外键，而是在应用层保证数据一致性。

用外键 不用外键
数据一致性 数据库保证 应用层保证
性能 写操作需检查约束，稍慢 更快
灵活性 删数据受约束 自由
分库分表 无法跨库 可以

面试时可以说：“项目中用了外键保证数据完整性，适合中小规模。如果数据量大到需要分库分表，外键会变成瓶颈，那时候在应用层用逻辑外键。”

9.6 六、常见问题排查

9.6.1 后端启动报 “port 8080 already in use”

→ 上次没关干净，任务管理器里找到 java.exe 杀掉，或重启电脑。

9.6.2 后端启动报 “Access denied for user ‘root’”

→ MySQL 密码不对。检查 application.yml 的 datasource.password。

9.6.3 前端报 “Cannot find module ‘element-plus’”

→ 依赖没装。在 news-web 目录下运行 npm install。

9.6.4 登录后页面空白

→ 按 F12 看 Console 报错。大概率是后端没启动或 API 路径不对。

9.6.5 注册后不能登录

→ 注册成功后需要手动切换到登录 tab 重新登录。注册不会自动登录。

9.6.6 修改代码后不生效

→ 前端：Vite 有 HMR（热更新），保存即生效。后端：需要重启（有 DevTools 的情况下会自动重启）。
